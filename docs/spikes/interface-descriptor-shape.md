# Interface Descriptor Shape Spike

Status: exploratory spike
Date: 2026-05-01

## Goal

Evaluate whether a generated artifact-local interface descriptor can carry the
information users need when binding a packaged plugin to a trigger, while
keeping the registry index focused on catalog metadata.

This spike assumes:

- The registry index remains a small discovery document: name, version,
  description, trigger entrypoints, dependencies, compatibility, hash, yank
  state, and publication time.
- Full interface metadata lives inside the downloaded artifact, for example at
  `.influxdb3/interface.json`.
- The interface descriptor is generated by the SDK from a Python authoring API,
  not hand-maintained separately from the plugin implementation.

Examples below use YAML for readability. The packaged artifact should emit
canonical JSON.

- `config_file_path` plus `PLUGIN_DIR` path resolution.
- Notification channel configuration shared by alerting plugins.
- Message subscriber mapping DSLs for MQTT, Kafka, and AMQP.
- HTTP request contracts that are not the same as trigger arguments.

## Descriptor Model

The descriptor should describe each plugin entrypoint as a callable interface.
Each callable can have multiple configuration surfaces:

- `trigger_arguments`: arguments supplied when the trigger is created.
- `request.query`: HTTP query parameters for `process_request`.
- `request.headers`: HTTP request headers for `process_request`.
- `request.body`: HTTP request body for `process_request`.

That split is necessary because HTTP plugins frequently have no trigger
arguments, but still require request-time input.

```yaml
interface_schema_version: "0.1"
triggers:
  process_scheduled_call:
    trigger_arguments: { ... }
  process_writes:
    trigger_arguments: { ... }
  process_request:
    trigger_arguments: { ... }
    request:
      query: { ... }
      headers: { ... }
      body: { ... }
```

The schema language should target JSON Schema 2020-12, with InfluxDB-specific
extensions for UI and runtime details that JSON Schema does not standardize.
Draft 2020-12 matters because `unevaluatedProperties` composes more cleanly
with `$ref` / `allOf` than older `additionalProperties`-only patterns.

Recommended extensions:

- `x-influxdb3-group`: Display grouping such as `connection`, `security`, or
  `notification`.
- `x-influxdb3-secret`: UI hint for sensitive values.
- `x-influxdb3-wire-type`: The runtime encoding when semantic type and wire
  type differ. Current trigger arguments are string maps, even when the logical
  value is a boolean or integer.
- `x-influxdb3-toml-path`: Equivalent TOML path for plugins that support config
  files.
- `x-influxdb3-format-doc`: Human-readable mini-language name when a value is a
  DSL rather than a primitive.
- `x-influxdb3-external-file`: Marks arguments that point at a file the user must
  place in the plugin directory.

## Common Definitions

These fragments recur across many plugins and should be reusable.

```yaml
$defs:
  TomlConfigRef:
    type: string
    description: Absolute path or path relative to PLUGIN_DIR.
    x-influxdb3-format-doc: file-path
    x-influxdb3-external-file: true

  DurationString:
    type: string
    pattern: "^[0-9]+(s|min|m|h|d|w|M|q|y)$"
    examples: ["30s", "5min", "1h", "30d"]

  DotSeparatedList:
    type: string
    description: Dot-separated list.
    examples: ["field_a.field_b.field_c"]

  Base64Json:
    type: string
    contentEncoding: base64
    description: Base64-encoded JSON.

  FilePath:
    type: string
    x-influxdb3-format-doc: file-path

  SecretString:
    type: string
    writeOnly: true
    x-influxdb3-secret: true
```

## Example 1: Simple Scheduled Args

Plugin: `system_metrics/system_metrics.py`

This is the simplest useful case. Every argument is optional and has a default.
The current Python metadata lists all fields, and the README adds the logical
types and defaults.

```yaml
interface_schema_version: "0.1"
triggers:
  process_scheduled_call:
    trigger_arguments:
      type: object
      additionalProperties: false
      properties:
        hostname:
          type: string
          default: localhost
          description: Hostname to tag all metrics with for system identification.
          examples: ["db-node-1"]
          x-influxdb3-wire-type: string
        include_cpu:
          type: boolean
          default: true
          description: Include CPU metrics.
          x-influxdb3-wire-type: string
        include_memory:
          type: boolean
          default: true
          description: Include memory metrics.
          x-influxdb3-wire-type: string
        include_disk:
          type: boolean
          default: true
          description: Include disk metrics.
          x-influxdb3-wire-type: string
        include_network:
          type: boolean
          default: true
          description: Include network metrics.
          x-influxdb3-wire-type: string
        max_retries:
          type: integer
          default: 3
          minimum: 0
          description: Maximum retry attempts on failure.
          x-influxdb3-wire-type: string
        config_file_path:
          $ref: "#/$defs/TomlConfigRef"
```

What this proves:

- Flat JSON Schema is enough for simple trigger arguments.
- Current string-map trigger arguments need either coercion rules or explicit
  `x-influxdb3-wire-type`.
- Defaults belong in the interface descriptor, not only in README prose.

## Example 2: Multiple Entrypoints

Plugin: `basic_transformation/basic_transformation.py`

This plugin supports both scheduled and write triggers. The entrypoints share
many arguments, but `window` only applies to the scheduled trigger. The
descriptor needs per-entrypoint signatures rather than one plugin-wide argument
bag.

```yaml
interface_schema_version: "0.1"
triggers:
  process_scheduled_call:
    trigger_arguments:
      unevaluatedProperties: false
      allOf:
        - $ref: "#/$defs/BasicTransformationCommonArgs"
        - type: object
          required:
            - measurement
            - target_measurement
            - window
          properties:
            window:
              $ref: "#/$defs/DurationString"
              description: Historical data window.
              examples: ["30d", "1h"]
              x-influxdb3-group: data-selection

  process_writes:
    trigger_arguments:
      unevaluatedProperties: false
      allOf:
        - $ref: "#/$defs/BasicTransformationCommonArgs"
        - type: object
          required:
            - measurement
            - target_measurement

$defs:
  BasicTransformationCommonArgs:
    type: object
    properties:
      measurement:
        type: string
        description: Source measurement containing data to transform.
        x-influxdb3-group: required
      target_measurement:
        type: string
        description: Destination measurement for transformed data.
        x-influxdb3-group: required
      target_database:
        type: string
        description: Database for storing transformed data. Defaults to the trigger database.
        x-influxdb3-group: required
      dry_run:
        type: boolean
        default: false
        description: Log transformations without writing.
        x-influxdb3-wire-type: string
      names_transformations:
        type: string
        description: Field/tag name transformation rules.
        examples: ["room:'lower snake'.temp:'upper camel'"]
        x-influxdb3-group: transformation
        x-influxdb3-format-doc: transformation-rule-list
      values_transformations:
        type: string
        description: Field value transformation rules.
        examples: ["temp:'convert_degC_to_degF'"]
        x-influxdb3-group: transformation
        x-influxdb3-format-doc: transformation-rule-list
      custom_replacements:
        type: string
        description: Custom string replacements.
        examples: ["rule_name:'find=replace'"]
        x-influxdb3-group: transformation
      custom_regex:
        type: string
        description: Regex patterns for field matching.
        examples: ["pattern_name:'temp%'"]
        x-influxdb3-group: transformation
      included_fields:
        $ref: "#/$defs/DotSeparatedList"
        description: Fields and tags to include.
        x-influxdb3-group: data-selection
      excluded_fields:
        $ref: "#/$defs/DotSeparatedList"
        description: Fields and tags to exclude.
        x-influxdb3-group: data-selection
      filters:
        type: string
        description: Query filters.
        examples: ["field:'operator value'"]
        x-influxdb3-group: data-selection
      config_file_path:
        $ref: "#/$defs/TomlConfigRef"
```

What this proves:

- The interface is per-entrypoint.
- Shared fragments reduce duplication, but the final emitted descriptor may need
  to be dereferenced if UI clients should not implement `$ref`.
- The old docstring format cannot represent grouping, defaults, semantic types,
  or scheduled-only/write-only differences cleanly.

## Example 3: HTTP Request Body Contract

Plugin: `notifier/notifier_plugin.py`

The plugin has no trigger arguments. Its real interface is the HTTP request
body accepted by `process_request`. Current plugin metadata says only
`plugin_type: ["http"]`, which is not enough for a user to call the endpoint.

```yaml
interface_schema_version: "0.1"
triggers:
  process_request:
    trigger_arguments:
      type: object
      additionalProperties: false

    request:
      body:
        type: object
        additionalProperties: false
        required:
          - notification_text
          - senders_config
        properties:
          notification_text:
            type: string
            description: Text content of the notification message.
          senders_config:
            type: object
            description: Map of notification channel name to channel configuration.
            minProperties: 1
            additionalProperties: false
            properties:
              slack:
                $ref: "#/$defs/SlackNotificationConfig"
              discord:
                $ref: "#/$defs/DiscordNotificationConfig"
              http:
                $ref: "#/$defs/HttpNotificationConfig"
              sms:
                $ref: "#/$defs/TwilioSmsConfig"
              whatsapp:
                $ref: "#/$defs/TwilioWhatsappConfig"

$defs:
  SlackNotificationConfig:
    type: object
    additionalProperties: false
    required: [slack_webhook_url]
    properties:
      slack_webhook_url:
        type: string
        format: uri
        x-influxdb3-secret: true
      slack_headers:
        type: string
        contentEncoding: base64
        description: Base64-encoded JSON headers.

  DiscordNotificationConfig:
    type: object
    additionalProperties: false
    required: [discord_webhook_url]
    properties:
      discord_webhook_url:
        type: string
        format: uri
        x-influxdb3-secret: true
      discord_headers:
        type: string
        contentEncoding: base64
        description: Base64-encoded JSON headers.

  HttpNotificationConfig:
    type: object
    additionalProperties: false
    required: [http_webhook_url]
    properties:
      http_webhook_url:
        type: string
        format: uri
        x-influxdb3-secret: true
      http_headers:
        type: string
        contentEncoding: base64
        description: Base64-encoded JSON headers.

  TwilioSmsConfig:
    type: object
    additionalProperties: false
    required:
      - twilio_from_number
      - twilio_to_number
    properties:
      twilio_sid:
        $ref: "#/$defs/SecretString"
        description: Twilio Account SID. May also come from TWILIO_SID.
      twilio_token:
        $ref: "#/$defs/SecretString"
        description: Twilio Auth Token. May also come from TWILIO_TOKEN.
      twilio_from_number:
        type: string
        description: Sender phone number in E.164 format.
      twilio_to_number:
        type: string
        description: Recipient phone number in E.164 format.

  TwilioWhatsappConfig:
    allOf:
      - $ref: "#/$defs/TwilioSmsConfig"
      - type: object
        description: Twilio WhatsApp sender and recipient numbers in E.164 format.
```

What this proves:

- HTTP request metadata is a separate context from trigger creation metadata.
- The descriptor needs to support nested objects.
- Secret fields and environment-variable fallbacks are UI/runtime annotations,
  not just prose.

## Example 4: Complex Conditional Interface

Plugin: `opcua/opcua.py`

OPC UA is the strongest stress test. It has mutually exclusive operating modes,
paired auth fields, security-policy dependencies, path/file handling, list-like
CLI values, regex fields, TOML section mapping, and defaults.

```yaml
interface_schema_version: "0.1"
triggers:
  process_scheduled_call:
    trigger_arguments:
      type: object
      additionalProperties: false
      required:
        - server_url
        - table_name
      oneOf:
        - title: Explicit nodes mode
          required: [nodes]
        - title: Browse mode
          required: [browse_root]
      properties:
        server_url:
          type: string
          description: OPC UA server endpoint URL.
          examples: ["opc.tcp://localhost:4840"]
          x-influxdb3-toml-path: opcua.server_url
          x-influxdb3-group: connection
        table_name:
          type: string
          description: InfluxDB measurement name for storing data.
          x-influxdb3-toml-path: opcua.table_name
          x-influxdb3-group: connection

        nodes:
          type: string
          description: Space-separated node mappings.
          examples: ["temperature:2:s=Temp:float pressure:2:s=Pressure"]
          x-influxdb3-format-doc: "field_name:namespace:identifier[:type]"
          x-influxdb3-toml-path: opcua.nodes
          x-influxdb3-group: explicit-nodes
        namespaces:
          type: string
          description: Space-separated namespace alias mappings.
          examples: ["siemens=urn:vendor:s7 beckhoff=urn:beckhoff:ua"]
          x-influxdb3-format-doc: "alias=namespace_uri"
          x-influxdb3-toml-path: opcua.namespaces
          x-influxdb3-group: explicit-nodes

        browse_root:
          type: string
          description: Root node ID for auto-discovery mode.
          examples: ["ns=2;s=Devices"]
          x-influxdb3-toml-path: opcua.browse.browse_root
          x-influxdb3-group: browse
        browse_depth:
          type: integer
          default: 2
          minimum: 1
          description: Maximum browse depth.
          x-influxdb3-wire-type: string
          x-influxdb3-toml-path: opcua.browse.browse_depth
          x-influxdb3-group: browse
        path_tags:
          type: array
          items: { type: string }
          default: []
          description: Object hierarchy levels to map to tags.
          x-influxdb3-wire-type: string
          x-influxdb3-toml-path: opcua.browse.path_tags
          x-influxdb3-group: browse
        filter:
          type: string
          format: regex
          description: Regex filter for discovered variable names.
          x-influxdb3-toml-path: opcua.browse.filter
          x-influxdb3-group: browse
        exclude_branches:
          type: string
          format: regex
          description: Regex for Object nodes whose subtree should be skipped.
          x-influxdb3-toml-path: opcua.browse.exclude_branches
          x-influxdb3-group: browse
        browse_tags:
          type: array
          items: { type: string }
          description: Variable names that should be stored as tags.
          x-influxdb3-wire-type: string
          x-influxdb3-toml-path: opcua.browse.browse_tags
          x-influxdb3-group: browse
        name_separator:
          type: string
          description: Separator for splitting variable names into tag segments.
          x-influxdb3-toml-path: opcua.browse.name_separator
          x-influxdb3-group: browse
        name_tags:
          type: array
          items: { type: string }
          description: Tag names extracted from leading name segments.
          x-influxdb3-wire-type: string
          x-influxdb3-toml-path: opcua.browse.name_tags
          x-influxdb3-group: browse

        default_tags:
          type: string
          description: Space-separated static tags added to every point.
          examples: ["location=factory_1 line=A"]
          x-influxdb3-format-doc: "key=value-list"
          x-influxdb3-toml-path: opcua.default_tags
          x-influxdb3-group: tags
        tag_nodes:
          type: string
          description: Space-separated tag node mappings.
          examples: ["machine_id:2:s=MachineID serial:2:s=SerialNumber"]
          x-influxdb3-format-doc: "tag_name:namespace:identifier"
          x-influxdb3-toml-path: opcua.tag_nodes
          x-influxdb3-group: tags
        quality_filter:
          type: string
          default: good
          description: Space-separated OPC UA quality categories to accept.
          examples: ["good", "good uncertain"]
          x-influxdb3-format-doc: "space-separated enum values: good, uncertain, bad"
          x-influxdb3-toml-path: opcua.quality_filter
          x-influxdb3-group: quality

        username:
          type: string
          description: Username for UserToken authentication.
          x-influxdb3-toml-path: opcua.auth.username
          x-influxdb3-group: auth
        password:
          $ref: "#/$defs/SecretString"
          description: Password for UserToken authentication.
          x-influxdb3-toml-path: opcua.auth.password
          x-influxdb3-group: auth

        security_policy:
          type: string
          enum:
            - Basic128Rsa15
            - Basic256
            - Basic256Sha256
            - Aes128Sha256RsaOaep
            - Aes256Sha256RsaPss
          x-influxdb3-toml-path: opcua.security.security_policy
          x-influxdb3-group: security
        security_mode:
          type: string
          enum: [Sign, SignAndEncrypt]
          default: SignAndEncrypt
          x-influxdb3-toml-path: opcua.security.security_mode
          x-influxdb3-group: security
        certificate:
          $ref: "#/$defs/FilePath"
          description: Path to client certificate in DER format.
          x-influxdb3-toml-path: opcua.security.certificate
          x-influxdb3-group: security
        private_key:
          allOf:
            - $ref: "#/$defs/FilePath"
            - $ref: "#/$defs/SecretString"
          description: Path to client private key in PEM format.
          x-influxdb3-toml-path: opcua.security.private_key
          x-influxdb3-group: security

        disable_config_cache:
          type: boolean
          default: false
          description: Reload configuration on every call instead of caching for one hour.
          x-influxdb3-wire-type: string
          x-influxdb3-toml-path: opcua.disable_config_cache
          x-influxdb3-group: advanced
        config_file_path:
          $ref: "#/$defs/TomlConfigRef"
          x-influxdb3-group: advanced

      allOf:
        - if:
            required: [username]
          then:
            required: [password]
        - if:
            required: [password]
          then:
            required: [username]
        - if:
            required: [security_policy]
          then:
            required:
              - certificate
              - private_key
        - if:
            required: [name_tags]
          then:
            required: [name_separator]
```

What this proves:

- JSON Schema can represent the important structural rules with `oneOf` and
  `if`/`then`.
- UI rendering will need extensions for groups, TOML paths, file-path handling,
  and secret fields.
- Some plugin arguments are better modeled as arrays or objects, but the current
  trigger-arguments wire format is a string map. The descriptor must decide
  whether UI clients coerce typed values into strings or the runtime evolves to
  accept typed JSON trigger arguments.

## Findings

1. The catalog/interface split is sound. The index should not carry full
   argument schemas.
2. The descriptor must model more than trigger arguments. HTTP request query
   parameters, headers, and bodies are first-class user-facing interfaces.
3. JSON Schema is expressive enough for the representative plugins if we allow
   small InfluxDB-specific annotation extensions.
4. The current docstring metadata is not expressive enough for correct trigger
   creation. It lacks semantic types, defaults, conditional requirements,
   secret markers, request body schemas, external file requirements, and
   grouping.
5. README content currently contains important contract details that should move
   into code-owned interface declarations.
6. The current trigger argument wire format is a limiting factor. Many values
   are semantically booleans, integers, arrays, objects, regexes, or file paths
   but are passed as strings.

## Risks And Open Questions

- Should `trigger_arguments` remain a string map, or should the database accept
  typed JSON values at trigger creation time?
- Does the database validate the descriptor at trigger creation, or is the first
  implementation UI/CLI-only validation?
- Should UI clients render `$ref` directly, or should the SDK emit a dereferenced
  descriptor for simpler consumers?
- Does the Python authoring API use Pydantic, dataclasses plus annotations, or a
  small SDK-owned schema DSL?
- Can the Rust SDK extract interface declarations statically, or does extraction
  require a controlled Python helper process?
- How should descriptor metadata reference external sidecar files that are
  required for correct operation, such as `schema_validator_config.json`?
- Should the artifact descriptor hash be recorded in the index as a future
  optional field, or is the artifact hash enough?

## Recommended Next Spike

Prototype the Python authoring API for two plugins:

- `system_metrics`, to validate the simple case.
- `opcua` or `mqtt_subscriber`, to validate conditional fields, groups, file
  paths, and TOML path annotations.

The goal should be author ergonomics, not full SDK integration. The output of
that spike is a concrete Python declaration style plus the generated descriptor
shape it can support.
