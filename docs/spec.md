# influxdb3-plugin-sdk Specification

This document contains the SDK-relevant portions of the plugin version management specification. It covers:

- **Spec 1** — Source & Distribution Contracts (manifest schema, index schema, artifact format, schema versioning, plugin identity)
- **Spec 2** — Plugin SDK (crate architecture, commands, output modes, validation, packaging, distribution, release engineering, requirements S2-1 through S2-21)

Sections covering database-side behavior (Spec 3: Registry & Resolution, Spec 4: Installation & State Management, Spec 5: Trigger Integration) are not included — they belong to the database runtime, not the SDK.

---

## Spec 1: Source & Distribution Contracts

**Purpose:** Define the data schemas that all other specs build on.
**Detail level:** Implementation-ready.
**Depends on:** —

### Manifest Schema

Format: TOML. Lives at the root of the plugin directory as `manifest.toml`. Author-owned; never generated.

```toml
manifest_schema_version = "1.1"

[plugin]
name          = "downsampler"              # validated: [a-zA-Z][a-zA-Z0-9_-]* (1-64)
version       = "1.2.0"                    # SemVer 2.0.0
description   = "Notify an HTTP endpoint on every WAL commit."
triggers      = ["process_writes", "process_scheduled_call"]
homepage      = "https://influxdata.com"                                     # optional
repository    = "https://github.com/influxdata/plugin-downsampler"           # optional
documentation = "https://github.com/influxdata/plugin-downsampler/readme.md" # optional

[dependencies]
database_version = ">=3.2.0,<4.0.0"       # SemVer range
python = [                                # PEP 508 strings; pip/uv-compatible
  "requests>=2.31,<3",
  "pydantic~=2.0",
]
```

**Field requirements:**

- `manifest_schema_version` (required) — Root-level string identifying the manifest format version. Parsed first so consumers can reject or dispatch on unsupported versions.
- `plugin.name` (required) — Validated against Cargo's crate-name rule `[a-zA-Z][a-zA-Z0-9_-]*` (1–64 ASCII characters, starting with a letter; case-preserving). Windows reserved device names (`con`, `prn`, `aux`, `nul`, `com0`–`com9`, `lpt0`–`lpt9`) are rejected case-insensitively. Uniqueness within a registry is checked on the canonical form `lowercase(name).replace('-', '_')`: two spellings that canonicalize equally (e.g. `foo-bar` and `foo_bar`, or `MyPlugin` and `myplugin`) are treated as the same plugin regardless of version — a registry cannot carry two distinct spellings of the same canonical name. Forms the second element of `(index_url, name, version)` identity.
- `plugin.version` (required) — SemVer 2.0.0 compliant.
- `plugin.description` (required) — One-line human-readable description, max 200 chars. Drives browse/discovery UX.
- `plugin.triggers` (required) — Non-empty array of trigger type identifiers the plugin implements.
- `plugin.homepage` (optional)
- `plugin.repository` (optional)
- `plugin.documentation` (optional)
- `dependencies.database_version` (required) — SemVer range specifier. Drives compatibility filtering in Stage 3.
- `dependencies.python` (optional) — Array of PEP 508 requirement strings. Empty or omitted means no Python dependencies.

**Reserved for future manifest schema versions:**
- `dependencies.plugins` — inter-plugin dependencies (decided deferred from v1). The current `plugin_library.json` schema has a `required_plugins` field but no plugin in the official repo populates it, so deferring this structured form has zero migration impact (see [[#What the index metadata contains today]]).

### Index Schema

Format: JSON. One index per registry. Generated by the packaging tool; never hand-edited.

```json
{
  "index_schema_version": "2.0",
  "artifacts_url": "https://plugins.influxdata.com/artifacts",
  "plugins": [
    {
      "name": "downsampler",
      "version": "1.2.0",
      "published_at": "2026-04-29T18:45:12Z",
      "description": "Notify an HTTP endpoint on every WAL commit.",
      "triggers": ["process_writes", "process_scheduled_call"],
      "homepage": "https://influxdata.com",
      "repository": "https://github.com/influxdata/plugin-downsampler",
      "documentation": "https://github.com/influxdata/plugin-downsampler/readme.md",
      "dependencies": {
        "database_version": ">=3.2.0,<4.0.0",
        "python": ["requests>=2.31,<3", "pydantic~=2.0"]
      },
      "hash": "sha256:9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08"
    }
  ]
}
```

**Top-level fields:**
- `index_schema_version` (required) — Version of the index format. Consumers refuse to parse unsupported majors.
- `artifacts_url` (required) — URL prefix under which all artifacts live. Combined with the flat naming convention `{artifacts_url}/{name}-{version}.tar.gz` to resolve artifact locations.

  **Supported schemes (v1):**
  - `https://` — recommended default. Covers S3 / GCS / Azure Blob object stores, GitHub raw / Pages / Releases, Artifactory / Nexus / Cloudsmith / Packagecloud, CDNs (CloudFront, Cloudflare, Fastly), and any TLS-fronted HTTP server. Credentials for private registries are supplied via registry config and applied at fetch time (e.g., AWS SigV4 presigning, Bearer tokens); they are never embedded in the index.
  - `http://` — plain HTTP. Intended for localhost development (`python -m http.server`, local packaging-tool iteration) and trusted internal networks. Integrity still rests on `hash`, but there is no transport-layer encryption. Consumers should warn when `http://` is used for a non-localhost registry.
  - `file://` — local filesystem. Used for offline / air-gapped deployments, iteration against the packaging tool's output directory, and appliance bundles that ship artifacts alongside the database binary.

  **Not supported in v1**
  - `oci://` and container-registry protocols — manifests-and-layers semantics don't match the flat artifact URL convention. Would require a different registry type, not just a different URL scheme.
  - `s3://` as a native resolver scheme — use the bucket's `https://` endpoint instead; SigV4 credentials live in registry config.
  - `git://`, `git+https://`, `git+ssh://` — Git is a source mechanism, not an artifact transport. Authors who iterate from Git produce a `tar.gz` via the packaging tool and serve it over `https://` or `file://`.
  - `ftp://`, `sftp://` — no current use case; `https://` covers the secure-transfer need.

  Unsupported schemes are rejected at registry-configuration time, not at fetch time, so bad configurations fail fast.
- `plugins` (required) — Flat array of per-version entries.

**Per-entry fields:**
- `name`, `version` (required) — Identity key pair within the registry. `(name, version)` is unique across the array.
- `published_at` (required) — Original publication timestamp for this exact plugin version, in Cargo registry-index `pubtime` format: an ISO 8601 UTC string with seconds precision and uppercase UTC designator, `YYYY-MM-DDTHH:MM:SSZ` (for example `2026-04-29T18:45:12Z`). Offsets, lowercase `z`, fractional seconds, leap seconds, and non-UTC forms are rejected.
- `description`, `triggers`, `dependencies.database_version`, `dependencies.python` (required) — Verbatim mirror of manifest fields. Lets the resolver filter, select, and resolve without fetching any artifact.
- `homepage`, `repository`, `documentation` (optional) — Verbatim mirror of the manifest's optional URL fields. Surfaced in browse/discovery so users can click through without downloading the artifact.
- `hash` (required) — SHA-256 of the published archive. Verified before extraction.
- `yanked` (optional) — Present and set to `true` when the version is yanked. Absence means not yanked.

**Index-entry validation mirrors manifest validation.** Index entries are derived from manifests and must satisfy the same field-level rules at parse time:
- `triggers` is non-empty and every entry is drawn from the closed set `{process_writes, process_scheduled_call, process_request}`.
- `homepage`, `repository`, `documentation` (when present) parse as valid URLs whose scheme is `http` or `https`.
- `dependencies.database_version` parses as a valid SemVer range.
- `dependencies.python` entries are PEP 508-parseable.
- `published_at` parses as strict Cargo-style UTC publication time (`YYYY-MM-DDTHH:MM:SSZ`) and represents a real calendar timestamp.

Consumers (the database resolver, the SDK's `validate --index`) reject index entries that violate any of these rules. This keeps "what the index says" and "what the manifest declared" in lockstep, preventing a publishing bug from leaking malformed metadata into a downstream install.

**Deliberately omitted from v2:**
- Per-entry download `url` — reconstructed from convention.
- Self-declared `registry` or `display_name` — the index cannot credibly attest to its own identity; registry identity lives in admin config.
- `generated_at`, `size`, `yank_reason` — audit/UX value only; not required for the core lifecycle.
- Per-entry `manifest_schema_version` — no current consumer branches on it; speculative forward-compat.
- Signatures / attestations — reserved under `signature` for a future schema version (additive).

**Registry layout is flat.** Artifacts are siblings at the same URL depth, not nested under `{name}/{version}/` directories. Browsability is not a goal — tooling enables discovery by reading the index.

### Artifact Format

- Archive format: gzipped tar (`tar.gz`). v1 supports `tar.gz` exclusively; no per-entry format selection. Migrating to another format (e.g., zstd) is an additive future-schema change — add an optional `format` field with `tar.gz` as the default.
- Contents: the full plugin directory, including `manifest.toml` and the plugin's Python source file(s). Files not needed for execution should be excluded.
- Integrity anchor: SHA-256 of the archive bytes, recorded in the entry's `hash` field. Verified before extraction. The hash is the sole trust anchor in v1.
- Immutability: once `(name, version)` is published, the artifact and its index entry (excluding `yanked` toggling) are immutable. The `published_at` value records original publication time and is preserved across derived indexes, including yank and unyank operations. Publishing the same `(name, version)` again is rejected.

**Artifact URL composition (flat):**
```
{artifacts_url}/{name}-{version}.tar.gz
```
Example: `https://plugins.influxdata.com/artifacts/downsampler-1.2.0.tar.gz`

Fully deterministic from `(artifacts_url, name, version)`; no per-entry URL is stored.

### Schema Versioning Strategy

Two independent schema versions evolve on their own cadence: `manifest_schema_version` (per manifest) and `index_schema_version` (per index). Decoupling means a registry can carry plugin entries authored against several manifest generations simultaneously.

**Evolution rules:**
- Within a schema's major version, fields may be added. Consumers ignore unknown top-level and per-entry fields.
- Breaking changes bump the major version. Consumers refuse to parse unsupported majors rather than guessing.
- Reserved keys (`dependencies.plugins`, `signature`, etc.) are named in v1 so third-party tools don't squat them.

**Index schema 2.0 migration guidance:** non-empty 1.x indexes must be backfilled with a required `published_at` field on every `plugins[]` entry before they can be parsed by v2 consumers. Exact historical publication timestamps are preferred. When exact history is unavailable, the registry operator must choose and document an appropriate fallback timestamp. This change does not ship an automatic migration script or SDK migration command; existing customers should migrate their index JSON as an operational data migration before adopting tooling that writes or requires index schema 2.0.

### Plugin Identity Model

Global plugin identity is the tuple `(index_url, name, version)`.

- **`index_url`** — the URL the admin configured when adding the registry. Never self-declared inside the index. Trust is anchored in the admin's decision to add the URL plus DNS/TLS control of the host.
- **`name`** — validated plugin name from the manifest, case-preserving. Uniqueness within a registry is enforced on the canonical form `lowercase(name).replace('-', '_')`, mirroring Cargo's `canon_crate_name`. A registry cannot carry two distinct spellings that share the same canonical form across any versions.
- **`version`** — SemVer 2.0.0 from the manifest. Unique for a given canonical `name` within a registry. Under the canonical-uniqueness rule above, every entry sharing a canonical name also shares the same spelling; different versions of the same plugin therefore share the same surface `name` exactly.

**Where identity surfaces:**
- **Registry config** (admin-side, cluster catalog): `index_url` is the identifier. An optional `local_alias` field is CLI convenience, local to the host, and is not part of identity.
- **Lockfile:** each installed plugin records `{ "source": { "registry": { "index_url": "…" } }, "name": "…", "version": "…", "artifact_hash": "…" }` for registry sources, or `{ "source": { "local": { "path": "…" } }, "name": "…", "version": "…" }` for local sources (no `artifact_hash`). Portable across databases that have configured the same registry. See Spec 4's [[#Lockfile Schema]].
- **Trigger definitions:** reference plugins by `(index_url, name, version)`.

**Why not a self-declared `registry` name or GUID:**
- An index cannot credibly attest to its own identity without a cryptographic signature. v1 has no signature layer, so a self-declared name is just a label — no stronger than what the URL already provides.
- A DB-minted GUID per-registry breaks lockfile portability across databases.
- URL-as-identity matches Cargo, npm, NuGet, pip/uv, and apt.

**Dependency confusion:** the same plugin name in two registries is unambiguously distinct because identities differ by `index_url`. Resolvers never silently substitute one for the other.

**Impersonation:** bottoms out in DNS + TLS ownership. Index signatures (Sigstore-style) are a future additive upgrade — `signature` is reserved for that.

### Yanking

- A version is yanked by setting `yanked: true` on its existing entry and regenerating the index. Entries are never deleted or renamed. Artifacts remain fetchable so existing lockfiles continue to resolve.
- `published_at` remains the original publication time when a version is yanked or unyanked.
- Yanking affects resolution only: the resolver skips yanked versions when selecting a version to install or upgrade. Already-installed plugins and active triggers continue to work.
- `yank_reason` is not carried in v1. If surfaced later it is an additive optional field.
- Authority to yank is operational: whoever controls the registry URL (DNS/TLS, write permissions on the backing store) controls yank state. Cryptographic author-level yank authority is out of scope for v1.

### Version Diff / Documentation Model

Deferred. No standard changelog or release-notes field in v1. Authors who need to surface version-to-version changes should ship them alongside the plugin source for now. Adding a `plugin.readme` or `plugin.changelog` field in a future manifest schema version is additive.

### Requirements

#### S1-1: Schema Version Fields

Both the plugin index and the plugin manifest must include a schema version field (`manifest_schema_version`, `index_schema_version`) so consumers can detect format incompatibilities and tools can negotiate versions during format evolution.

Problems: [[#No separation between plugin manifest and index]]

#### S1-2: Index Is Machine-Generated

The index is produced by the packaging tool from published manifests and artifacts. Hand-editing is unsupported. The generator is the sole writer. Regeneration is deterministic given identical inputs plus the publication timestamp chosen for any newly added entry; production packaging defaults that timestamp to current UTC.

Problems: [[#No separation between plugin manifest and index]]

#### S1-3: Artifact Integrity Hash

Every index entry must carry `hash` (SHA-256 of the archive). Consumers must verify the hash before extraction and reject mismatches.

Problems: [[#No artifact integrity verification]]

#### S1-4: Immutable Publication

Once `(name, version)` is published to a registry, the artifact bytes and the entry's non-yank fields are immutable. The only permitted mutation to an existing entry is toggling the `yanked` field.

Problems: [[#No published-version lifecycle controls]]

#### S1-5: Global Identity Is `(index_url, name, version)`

The plugin management system must identify plugins by the `(index_url, name, version)` tuple. `index_url` is sourced from admin-side registry configuration, not from the index itself.

Problems: [[#No plugin identity model]], [[#No global plugin identity across registries]]

#### S1-6: Flat Artifact Layout

Artifacts are addressed by the flat naming convention `{artifacts_url}/{name}-{version}.tar.gz`. Per-plugin and per-version directory nesting is not used.

Problems: [[#Plugins can only be hosted on GitHub]]

### Invariants

1. `(name, version)` is unique across `plugins[]`. Additionally, the canonical form of `name` (`lowercase(name).replace('-', '_')`) is unique across `plugins[]` regardless of version: two distinct spellings that canonicalize equally cannot coexist in one registry.
2. `hash` matches the published bytes exactly.
3. Yanking toggles the `yanked` field; entries are never deleted and `published_at` is preserved.
4. Generator is the sole writer; no hand edits.
5. Archive regeneration is deterministic given identical package inputs. Derived-index regeneration is deterministic given identical inputs plus publication time; production package defaults publication time to current UTC.
6. Unknown top-level and per-entry fields are ignored within a schema major; breaking changes bump the major.
7. `artifacts_url` supports the v1 scheme set enumerated in the Index Schema section (`https://`, `http://`, `file://`). Unsupported schemes (`oci://`, `s3://`, `git://`, `ftp://`, etc.) are rejected at registry-configuration time.
8. Registry identity lives in admin-side config as `index_url`; the index never self-declares.

## Spec 2: Plugin SDK

**Purpose:** Define the standalone CLI tooling that authors use to create, validate, package, and prepare plugins for publishing.
**Detail level:** Design-level.
**Depends on:** Spec 1

### Distribution & Crate Structure

The SDK ships as a separate Rust-workspace repository (`influxdb3-plugin-sdk`), producing a standalone `influxdb3-plugin` binary in v1. The crate layout is designed so a later phase can embed it as a `plugin` subcommand of the main `influxdb3` CLI without refactoring.

**Repository layout:**

```
influxdb3-plugin-sdk/
├── Cargo.toml                      # workspace root
└── crates/
    ├── influxdb3-plugin-schemas/   # Manifest, IndexEntry, PluginId types + parsers.
    │                               # Zero I/O, zero clap, tiny public API.
    ├── influxdb3-plugin-sdk/       # Library: scaffold, package, validate, hash,
    │                               # mutate index. Depends on schemas.
    └── influxdb3-plugin-cli/       # Binary `influxdb3-plugin` + public
                                    # `#[derive(clap::Parser)] struct PluginConfig`.
```

**Why three crates.** `schemas` is the canonical home for manifest/index types shared between the SDK and the db runtime — the runtime also parses manifests at install time (Spec 4) and reads index entries at resolve time (Spec 3). A shared schemas crate prevents drift between author-side and runtime-side types. `sdk` is the author-side library; `cli` wraps it in clap-derived commands and produces the binary.

**Phase plan:**
- **Phase 1 (v1 — scope of this spec).** Authors install and invoke the standalone `influxdb3-plugin` binary. The SDK does not integrate into `influxdb_pro`.
- **Phase 2 (out of v1 scope; forward-compatibility goal only).** `influxdb_pro` may later add a dependency on `influxdb3-plugin-cli` and mount `PluginConfig` as a variant of its top-level `Command` enum, following the existing `influxdb3_commands::query::Config` pattern. If and when that ships, both entry points coexist and share the same `sdk` crate code. Whether to do this is a later decision. The v1 SDK is designed not to foreclose it — see [[#Phase-2 Embedding Constraints]].

**Publishing model.** All three crates publish from v1 (crates.io or a private registry) so downstream consumers — including a hypothetical phase-2 `influxdb_pro` integration — can consume them by version without submodule or repo-merge coupling.

**Language.** Rust. Ships as a single compiled binary. Plugin *runtime* remains Python; the SDK tooling is Rust.

### Design Decisions

Implementation choices that refine Spec 2's structural requirements. Recorded here so downstream implementation plans, future maintainers, and cross-workspace consumers do not re-derive them. These decisions strike a balance between idiomatic Rust conventions and InfluxData org patterns established in `influxdb_pro`; deviations are called out with rationale.

#### Workspace conventions

- **Flat top-level crate layout.** The three member crates — `influxdb3-plugin-schemas`, `influxdb3-plugin-sdk`, `influxdb3-plugin-cli` — live directly under the workspace root, not under a `crates/` subdirectory. Matches the convention in `influxdb_pro/ent/`. At three members, `crates/` nesting adds no navigation value.
- **Hyphenated crate names.** `influxdb3-plugin-schemas` etc. **Diverges from `influxdb_pro`'s snake_case.** Chosen to match crates.io convention for externally-published tooling consumed by plugin authors outside InfluxData. Rust import names derive automatically (`influxdb3_plugin_schemas`).
- **`resolver = "3"`.** Edition 2024's default. **Diverges from `influxdb_pro`'s `resolver = "2"` carry-over.** Resolver 3 improves feature unification; no compatibility concern for a new workspace with no legacy dependencies.
- **Workspace inheritance.** `[workspace.package]` holds `version`, `authors`, `edition`, `rust-version`. `[workspace.dependencies]` centralizes dependency versions. `[workspace.lints]` defines lint policy. Per-crate `Cargo.toml` files opt in via `.workspace = true` everywhere and `[lints] workspace = true`. Matches `influxdb_pro` verbatim.
- **`default-members = ["influxdb3-plugin-cli"]`.** `cargo build` from workspace root builds the primary binary by default.
- **`license = "MIT OR Apache-2.0"` declared in `[workspace.package]`.** Matches `influxdb_pro/ent/Cargo.toml` and the standard Rust dual-license convention. `LICENSE-APACHE` and `LICENSE-MIT` are committed at the workspace root and copied verbatim from the org-standard license texts.
- **MPL-2.0 transitive exception.** `pep508_rs -> pep440_rs -> version-ranges` pulls in MPL-2.0; allowed via `[licenses.exceptions]` in `deny.toml`, matching `influxdb_pro`'s pattern for `webpki-roots` / `colored`. This is a permanent transitive-license exception, not a bridging measure. **Cross-repo note:** when `influxdb3-plugin-schemas` is first consumed inside `influxdb_pro/ent` (Spec 3/4 integration), the same `{ name = "version-ranges", allow = ["MPL-2.0"] }` entry must be added to `ent/deny.toml`; until then, adding it preemptively produces an `unused-allowed-license` warning with no matching dependency path.

#### Lints

Port `influxdb_pro`'s `[workspace.lints]` tables verbatim, minus `future_not_send`. The SDK has no concurrent execution story — `influxdb3-plugin-schemas` and `influxdb3-plugin-sdk` are synchronous libraries, and `influxdb3-plugin-cli`'s `PluginConfig::run()` is async per [[#S2-4 Async Entry Point|S2-4]] but does not spawn. `future_not_send` produces no useful signal in this context.

```toml
[workspace.lints.rust]
missing_copy_implementations = "deny"
missing_debug_implementations = "deny"
rust_2018_idioms = { level = "deny", priority = -1 }
unreachable_pub = "deny"
unused_crate_dependencies = "deny"

[workspace.lints.clippy]
clone_on_ref_ptr = "deny"
dbg_macro = "deny"
todo = "deny"

[workspace.lints.rustdoc]
bare_urls = "deny"
broken_intra_doc_links = "deny"
```

#### Profiles

Default `dev` and `release` only, plus one addition: `[profile.dev.package.insta] opt-level = 3`, per insta's own documentation — measurable dev-loop speedup for a one-line cost. **Skip `influxdb_pro`'s `quick-release`, `quick-bench`, `check-fast`, and `[profile.release] lto = "fat"`** — those earn their keep on a 60-crate database server build, not on a 3-crate CLI workspace.

#### Testing conventions

- **Inline test modules.** Tests live in `#[cfg(test)] mod tests { ... }` blocks inside their production file, not in sibling `*_tests.rs` files. **Diverges from `influxdb_pro`'s external-test-module convention**, and `scripts/check-test-modules.sh` is intentionally not ported.

  Rationale: the SDK's audience is external plugin authors and AI-assisted contributors who default to inline `#[cfg(test)]` per the Rust Book and mainstream ecosystem convention. The org's sibling-file convention adds friction without payoff for an externally-facing workspace.

- **Doctests on top-level public API.** Minimal doctests on entry-point APIs (`Manifest::parse_toml`, `Index::parse_json`, key `PluginConfig` methods). **Diverges from `influxdb_pro`'s no-doctest culture.**

  Rationale: `influxdb3-plugin-schemas` and `influxdb3-plugin-cli` are semver-stable published crates (see [[#Stability & Compatibility]]). docs.rs renders doctests as runnable examples; absence of any doctest on a published crate signals low effort to ecosystem consumers.

#### `influxdb3-plugin-schemas` crate specifics

- **Module layout (flat):**
  - `src/lib.rs` — public re-exports, crate-level rustdoc, module declarations only
  - `src/error.rs` — `SchemaError` enum
  - `src/identity.rs` — `PluginId`, `PluginName`
  - `src/manifest.rs` — `Manifest`, `PluginMetadata`, `Dependencies`, `TriggerType`, `Description`, `ManifestSchemaVersion`, `PythonRequirement`
  - `src/index.rs` — `Index`, `IndexEntry`, `IndexSchemaVersion`, `ArtifactsUrl`, `ArtifactHash`

- **Error model.** A single `SchemaError` enum built with `thiserror`, annotated `#[non_exhaustive]` so variant additions are minor-version changes per [[#Stability & Compatibility]]. Each variant carries structured fields (name, version, scheme, etc.) for programmatic inspection, plus `#[source]` chaining to upstream parser errors (`semver::Error`, `toml::de::Error`, `serde_json::Error`, `pep508_rs::Pep508Error`, `url::ParseError`). Each variant's `Display` shape is pinned by a snapshot test.

  **Two-phase parsing.** `Manifest::parse_toml` and `Index::parse_json` return `Result<Self, SchemaErrors>` where `SchemaErrors(Vec<ReportedError>)` wraps a collected error list, and each `ReportedError` carries a `FieldPath` locating the error inside the parsed document (e.g., `plugin.name`, `plugins[3].dependencies.python[0]`). Parsing proceeds in two phases:

  1. **Raw deserialize.** Crate-private `Raw*` structs with serde-only field types (`String`, `Vec<String>`, `bool`, `Option<String>`) accept any TOML/JSON document conforming to the field *shape*. TOML/JSON *syntax* errors (unbalanced brackets, missing commas) and required-field-missing errors surface at this phase as a single `SchemaError::TomlParse` / `JsonParse` and short-circuit further processing — serde cannot continue past a syntax error.
  2. **Structured validation and collection.** Once raw deserialization succeeds, a validation pass walks the raw struct, runs each field validator (name regex, SemVer, trigger closed-set, URL scheme, PEP 508, hash format, description length), and collects every error along with its field path into `Vec<ReportedError>`. All errors from a single document are returned together; the user sees every defect in one pass.

  **Schema-version short-circuit.** If `manifest_schema_version` / `index_schema_version` parses to an unsupported major or is malformed, the parser returns exactly one error (`UnsupportedManifestMajor` / `UnsupportedIndexMajor` / `MalformedSchemaVersion`) and does not attempt to validate the rest of the document. This matches Spec 1's Schema Versioning Strategy ("Consumers refuse to parse unsupported majors rather than guessing").

  **External Deserialize impls preserved.** The newtype types (`PluginName`, `Description`, `TriggerType`, `PythonRequirement`, `ArtifactsUrl`, `ArtifactHash`, `ManifestSchemaVersion`, `IndexSchemaVersion`) retain their custom `Deserialize` impls for consumers who embed them in their own serde types. The schemas crate's own `parse_toml` / `parse_json` no longer route through these impls — the Raw path uses `String` for every field validated elsewhere.

- **Canonical serialization via dedicated method.** `Index::to_canonical_json(&self) -> Result<String, SchemaError>` — not the default `Serialize` trait impl — encapsulates field ordering, plugin sorting by `(name, version)` with SemVer precedence, 2-space indent, NFC Unicode normalization, and trailing newline. A dedicated method gives full control over the byte-deterministic output required by [[#S2-3 Deterministic Regeneration|S2-3]] without constraining arbitrary `serde_json::to_string` callers.

- **Dependencies:** `serde` (derive), `toml`, `serde_json`, `semver` (with serde), `url` (with serde), `pep508_rs`, `unicode-normalization`, `thiserror`.

- **`pep508_rs` pre-1.0 caveat.** `pep508_rs` is at v0.9 at time of writing — any minor bump may reshape its API. `SchemaError::InvalidPythonRequirement`'s `source` field is typed as `Box<pep508_rs::Pep508Error<pep508_rs::VerbatimUrl>>`, so a breaking `pep508_rs` change forces a major bump of `influxdb3-plugin-schemas`. Document this risk in the variant's rustdoc; recommend consumers prefer `.source()` (type-erased) over pattern-matching the typed `source` field.

#### `influxdb3-plugin-sdk` crate specifics

- **Python source analysis uses tree-sitter, not pyo3.** The Spec 2 [[#Validation]] code/manifest cross-reference (`__init__.py` parses as Python 3; top-level synchronous `def <trigger>()` exists; `async def` rejected) is performed via `tree-sitter` + `tree-sitter-python` (both MIT on crates.io). The database runtime uses pyo3 via `influxdb3_py_api` to *execute* plugin code; the SDK uses tree-sitter to *statically analyze* it at author-time. These are different operations on the same input:
  - pyo3 at SDK-time would force libpython linking at build + runtime, complicating the cross-compile matrix (S2-19's 4 targets) and adding a runtime dep the author's machine may not satisfy uniformly.
  - Shelling out to `python3` at SDK-time would delegate to the system interpreter — good accuracy, bad DX (a Rust SDK binary that fails with "python3 not found" is a confusing setup error for a tool that reads source files).
  - tree-sitter ships its parser as C code linked into the SDK binary. Zero runtime deps, single static binary per target, standard Rust cross-compile.
- **Candidates ruled out, with rationale (for future reference):**
  - `ruff_python_parser` — not published to crates.io; Astral has explicitly declined to publish implementation crates.
  - `rustpython-parser` — pulls `malachite` (LGPL-3.0) + multiple unmaintained-crate advisories (`paste`, `unic-*`).
  - `enderpy_python_parser` — AGPL-3.0-only; viral copyleft unsuitable for a consumer-facing SDK.
  - Hand-rolled tokenizer — Python string/indentation handling is nontrivial enough that a bug-free implementation isn't worth the dep-size savings.

### Phase-2 Embedding Constraints

Phase-2 embedding of the SDK into `influxdb_pro`'s CLI is not a v1 deliverable, but several v1 design constraints on the SDK keep that door open. A throwaway spike validated the embedding mechanism end-to-end against a local copy of `influxdb_pro`; the findings below derive from that exercise.

**Validated constraints the SDK MUST satisfy in v1:**

- **Async dispatch.** `PluginConfig::run()` is `pub async fn run(self) -> anyhow::Result<()>`. The host's non-serve dispatch already executes inside a `tokio::block_on` scope, so `config.run().await` embeds cleanly without a nested runtime. See [[#S2-4 Async Entry Point|S2-4]].
- **Self-declared version.** `PluginConfig` carries a `#[command(version = …)]` attribute sourced from crate metadata; the resulting output shape is pinned by [[#S2-21 --version Output Shape|S2-21]]. Without it, `influxdb3 plugin --version` errors because clap does not inherit `--version` across subcommand boundaries. Validated: override renders the SDK's version string while host `influxdb3 --version` is unchanged. See [[#S2-5 Self-Declared Version|S2-5]].
- **Clap version range matches the host's resolved clap.** `influxdb_pro`'s nominal workspace pin is permissive (`clap = "4"`), but transitive deps enforce a real floor (currently `^4.5.47`, set by `clap_builder`). The SDK's clap pin must stay at or above that floor; below it, Cargo resolution fails. The SDK's pin can also *downgrade* the host's clap if the host is willing to unify downward, which is reversible but disruptive. See [[#S2-6 Clap Version Range Policy|S2-6]].
- **No `std::process::exit`.** All error paths return through `Result`. The host's dispatch converts the error to a standard exit code and ensures tokio runtime shutdown, trace flushing, and panic-hook cleanup happen in order. See [[#S2-7 No Direct Process Exit|S2-7]].
- **No process-global-state installation from the library surface.** The SDK library does not install a `#[global_allocator]`, TLS/crypto providers, nested tokio runtimes, tracing subscribers or log backends, panic hooks, signal handlers, or OpenTelemetry providers. These are host-owned concerns. The SDK's standalone binary entry point (`main.rs`) is free to install these for standalone-mode use only; the library surface consumed during embedding must stay clean. See [[#S2-8 No Global-State Installation|S2-8]].
- **Env-var namespace.** All `env = "…"` clap bindings in the SDK use the `INFLUXDB3_PLUGIN_*` prefix. See [[#S2-9 Env-Var Namespace|S2-9]].
- **Shared schema types.** Manifest, IndexEntry, and related parse-time types live in the `influxdb3-plugin-schemas` crate. Both the SDK and any future db-runtime plugin-management code consume the same versioned crate. Prevents drift of parse-time vs install-time types. See [[#S2-10 Schema Types from Shared Crate|S2-10]].

**Known open risk deferred to phase 2 (not a v1 concern):**

- **Host help renderer coupling.** `influxdb_pro`'s `influxdb3/src/lib.rs` hand-rolls its top-level help via `maybe_print_help()`, which intercepts `--help` and `--help-all` at every depth, not just root. An embedded subtree would have no clap-native help UX — `influxdb3 plugin new --help` would render the host's top-level help, not the SDK's subcommand help. Resolving this requires the host to let `--help` flow through clap once past the top-level command, OR the SDK to export a help-fragment contract that the host's renderer composes. Out of v1 scope because v1 does not embed. Must be addressed before phase 2 ships. Flagged here so that the v1 SDK's public surface does not foreclose either solution — adding a `help_fragment()` accessor later is additive.

### Stability & Compatibility

The three SDK crates have distinct stability policies that bind at each crate's own `1.0.0` release. During `0.x`, Cargo's SemVer convention permits breaking changes at any minor bump; the policies below describe the contract that engages at `1.0.0` and after:

- **`influxdb3-plugin-cli`** — stable embedding contract. The public API — `PluginConfig`, its subcommand enum, `pub async fn run(self) -> anyhow::Result<()>`, clap attribute surface (arg names, env-var bindings, version declaration), schema-type re-exports, and the JSON output schema emitted by every CLI command in `--output json` mode — is covered by semver. Breaking changes require a major-version bump. Requirements S2-4 through S2-10 name the load-bearing elements of this contract; S2-16 extends it to the JSON output shape.
- **`influxdb3-plugin-schemas`** — stable versioned types. Manifest, IndexEntry, and related parse-time types follow semver independently; their crate-semver tracks [[#Schema Versioning Strategy]]. Additive fields within a schema major are minor bumps; breaking field changes bump the crate's major.
- **`influxdb3-plugin-sdk`** — internal. Not a supported consumer surface; its API may change without notice. Consumers who need programmatic access to SDK functionality go through `influxdb3-plugin-cli`'s public interface, not directly through `sdk`. This keeps refactoring freedom in the layer where the most iteration is expected.

Rationale: the stability commitment is concentrated at the two boundaries that actually cross crate boundaries in practice — the embedding surface (`cli`) and the shared type layer (`schemas`). Scoping it this way prevents the internal `sdk` crate from becoming a de facto frozen API before its design has settled.

### Commands

The v1 SDK ships four commands plus a top-level `--version` flag. All operate on local files; none make network calls (S2-1). The CLI surface is deliberately minimal in v1 — no short flag forms beyond clap's default `-V`/`-h`, no env-var bindings, no verbose/quiet modes. Exit codes follow the GNU baseline: `0` success, `1` runtime failure, `2` usage error (per [[#S2-18 Exit Code Set|S2-18]]). Every command supports `--output {human,json}` per [[#Output Modes]]; additional machine formats (`sarif`, `yaml`, `github`, `gitlab`) are reserved post-v1 and add through the same enum. These can be added later without breaking v1 behavior. `new` additionally has a `list` subcommand and per-template help surfaces; see its section for details.

#### `new <template> [path]`

Scaffold a new plugin or index from a compile-time built-in template, or use the `list` subcommand to see available templates.

**Usage:**
- `influxdb3-plugin new <template> [path]` — scaffold the named template into `[path]`.
- `influxdb3-plugin new list` — print the available templates.
- `influxdb3-plugin new <template> -h` — print the flags (global and per-template) for a specific template.

**Built-in templates (v1):**

| Template name          | Short name               | Description                                             |
| ---------------------- | ------------------------ | ------------------------------------------------------- |
| Process Writes Plugin  | `process_writes`         | Plugin triggered by rows written to a database.         |
| Scheduled Call Plugin  | `process_scheduled_call` | Plugin triggered on a schedule.                         |
| Process Request Plugin | `process_request`        | Plugin triggered by an HTTP request.                    |
| Index                  | `index`                  | Empty registry index file.                              |

Each template carries metadata (template name, short name, description) and its own set of flags. Templates are compile-time built-ins in v1. The CLI is structured so that a new template is added as a self-contained module plus a single registration entry in the central list of built-in templates; no other CLI code changes are required. User-extensible templates (install / uninstall / search) are explicitly deferred post-v1.

**Positional arguments:**
- `<template>` (required) — the short name of a built-in template. An unknown short name exits with code 2 (usage error) and a message pointing at `new list`.
- `[path]` (optional) — target directory. Defaults to `.` (current working directory). Created if it does not exist.

**Global flags (every `new <template>` invocation):**
- `--output <mode>` — output format; one of `human` (default on TTY) or `json` (default when stdout is not a terminal or `CI=true`). See [[#Output Modes]].
- `--force` — allow the scaffold to overwrite files the selected template would write if they already exist. Without `--force`, the command fails if any file in the template's write set already exists. Files in the target directory that the template does not write are not consulted, regardless of `--force`.

**Per-template flags:**

Plugin templates (`process_writes`, `process_scheduled_call`, `process_request`):
- `--name <name>` — plugin name written into the scaffolded manifest. Defaults to the basename of `[path]`. If the derived name does not match the plugin-name rule (`[a-zA-Z][a-zA-Z0-9_-]*`, 1–64 ASCII chars starting with a letter; Windows reserved device names rejected case-insensitively), the command errors and asks for an explicit `--name`. Mirrors cargo's invalid-name behavior.
- `--database-version <range>` — SemVer range written into the scaffolded manifest as `dependencies.database_version`. Defaults to `>=<INFLUXDB3_PLUGIN_SDK_KNOWN_LATEST_DB>`, a value baked into the SDK at build time (see [[#Release Engineering]]).

Index template (`index`):
- `--artifacts-url <url>` — URL written into the generated `index.json` as the registry's `artifacts_url`. An explicit value is written verbatim (subject to scheme validation). When omitted, defaults to `file://<absolute path of [path], with symlinks preserved>`, making a freshly scaffolded local registry immediately consumable as a `file://` registry.

`new <template> -h` prints the template's description, its positional arguments, the global flags, and only the per-template flags that apply to that template. The top-level `new --help` does not enumerate templates; `new list` does.

**`new list`:**
- Prints a two-column table (template name, short name) in human mode.
- Emits `{ "templates": [ { "name": ..., "short_name": ... }, ... ] }` in json mode.
- Does not touch the filesystem. Supports `--output` only; passing `--force` to `list` exits 2 (usage error) — the flag is deliberately not declared on a read-only subcommand.
- Descriptions are not included in `new list` output — they appear in `new <template> -h`. Filtering (by language, tag, author) and template-package details are explicitly deferred post-v1.

**Outputs:**
- Plugin templates write `manifest.toml`, `__init__.py`, and a `README.md` stub into the target directory. The scaffolded `manifest.toml` bakes in the current `manifest_schema_version`; `__init__.py` wires the entrypoint for the selected trigger type.
- Registry template writes `index.json` into the target directory, baking in the current `index_schema_version`.

**Conflict behavior:**
- Without `--force`: compute the exact set of files the selected template will write, then fail if any of those already exist. Files the template does not write are left alone.
- With `--force`: the template's write set overwrites any conflicting files. Unrelated files in the target directory are still left alone.
- All-or-nothing: partial writes are never persisted; the command either writes its full file set or nothing.

#### `validate [plugin-dir]`

Validate a plugin directory without producing an artifact or modifying any index. Useful for PR-check CI flows that want a fast manifest/schema check without the cost of bundling.

**Positional arguments:**
- `[plugin-dir]` (optional) — plugin directory to validate. Defaults to `.`.

**Flags:**
- `--output <mode>` — output format; one of `human` (default on TTY) or `json` (default when stdout is not a terminal or `CI=true`). See [[#Output Modes]]. `sarif` is reserved for v1.1 — diagnostics are structured now to accommodate it.
- `--index <path>` (optional) — input index. When provided, both the `(name, version)` immutability check and the canonical-name uniqueness check (S2-2) run against it. When omitted, uniqueness is not checked — useful for pre-merge validation where the target index is not yet available or not yet relevant.

**Behavior:**
- Runs the checks listed in [[#Validation]]. All validation errors are collected and reported together rather than failing on the first.
- Produces no output files. Writes nothing. Exits 0 on valid, 1 on failure.
- The first half of `package` — authors who run `validate` then `package` will see identical results from the validation phase.

#### `package [plugin-dir]`

Validate a plugin directory, bundle it into an artifact, and emit a derived index containing the new entry. See also [[#Packaging]].

**Positional arguments:**
- `[plugin-dir]` (optional) — plugin directory to package. Defaults to `.`.

**Flags:**
- `--output <mode>` — output format; one of `human` (default on TTY) or `json` (default when stdout is not a terminal or `CI=true`). See [[#Output Modes]].
- `--index <path>` (required) — input index. Read-only.
- `--out <dir>` (required) — output directory. The derived index and the new artifact both land here.

**Behavior:**
- Runs validations per [[#Validation]]. All validation errors are collected and reported together rather than failing on the first — authors fix everything in one pass.
- Rejects the operation when `(name, version)` already exists in the input index (S2-2, immutable publication).
- Writes `<out>/index.json` (input index with the new entry appended) and `<out>/<name>-<version>.tar.gz`.
- Errors if `<out>` resolves to the directory containing `--index`. The input is always preserved (S2-11, S2-12).

#### `yank <name>@<version>`

Toggle the `yanked` field on an existing index entry, producing a derived index. No artifact is produced.

**Positional arguments:**
- `<name>@<version>` (required) — at-separated identifier of the entry to toggle (cargo-style).

**Flags:**
- `--output <mode>` — output format; one of `human` (default on TTY) or `json` (default when stdout is not a terminal or `CI=true`). See [[#Output Modes]].
- `--index <path>` (required) — input index. Read-only.
- `--out <dir>` (required) — output directory. The derived index is written to `<out>/index.json`.
- `--undo` (optional) — inverts the operation: clears `yanked` instead of setting it.

**Behavior:**
- Idempotent when the target is already in the desired state. Re-yanking an already-yanked version, or `--undo`-ing a not-yanked version, exits successfully with an informational message.
- Errors if `<name>@<version>` is not present in the input index.
- Subject to the same `--out` / `--index` non-overlap constraint as `package` (S2-11, S2-12).

#### `--version`

Top-level flag (not a subcommand). Prints a single line containing the SDK's version (from `CARGO_PKG_VERSION`) and the git commit hash from which the binary was built:

```
influxdb3-plugin <version>, revision <sha>
```

Example: `influxdb3-plugin 1.0.0, revision a5ed19d8e3f4c2b1a09f8e6d5c4b3a2918f7e6d5c`.

The format matches the `influxdb3` binary's `build_version_string` (`{product}, {version}, revision {sha}`, defined in `oss/influxdb3_process/src/lib.rs` in the `influxdata/influxdb_pro` repo); when the SDK is embedded as `influxdb3 plugin --version`, the output is visually consistent with the host's top-level `--version`.

The `<sha>` is sourced from (in precedence): the `GIT_HASH` env var when set, then `.cargo_vcs_info.json` at the crate root (Cargo's publish-time SHA capture, written by `cargo publish` and shipped inside the `.crate` tarball), then `git rev-parse HEAD`, with a final fallback to the literal `unknown`. The `cargo install` and source-tarball cases are covered by `.cargo_vcs_info.json`, so the dominant non-CI path produces a real SHA — `unknown` only fires for genuinely uncontrolled rebuilds (Homebrew, Nix, hand-unpacked tarballs rebuilt without `cargo install`).

A build date is not included. Per-build dates would defeat binary reproducibility; the SHA alone identifies the source commit unambiguously.

`--version` is exempt from `--output` per [[#Output Modes]] — it always prints plain text regardless of mode.

See [[#S2-5 Self-Declared Version|S2-5]] for the clap attribute requirement and [[#S2-21 --version Output Shape|S2-21]] for the binding format commitment.

### Output Modes

Every SDK command supports two output modes in v1, selected by `--output <mode>`. The mode set is `{human, json}`. The flag shape is an extensible enum so that additional machine formats (`sarif` on `validate`, `yaml`, `github`, `gitlab`) can be added post-v1 as additive changes without introducing new flags.

**`human` (default on TTY)** — colorized, human-readable rendering. Intended for interactive CLI use by plugin authors. Formatting details are not part of the stability contract; consumers must not parse human output.

**`json` (default in CI / when stdout is not a terminal)** — machine-readable JSON on stdout with the stable schema defined in [[#S2-16 JSON Output Schema Stability|S2-16]]. Intended for CI pipelines and scripted automation.

#### Auto-detection

When `--output` is omitted, the SDK selects the mode by these rules, evaluated in order:

| Condition | Mode |
|---|---|
| `--output <mode>` passed explicitly | That mode |
| `!isatty(stdout)` (stdout is piped, redirected, or going to a CI runner that does not attach a TTY) | `json` |
| `CI=true` environment variable is set to a truthy value (`true`, `1`) | `json` |
| Otherwise | `human` |

Detection uses Rust stdlib `IsTerminal` only; platform-specific CI env vars (`GITHUB_ACTIONS`, `GITLAB_CI`, `JENKINS_URL`, `BUILDKITE`, `CIRCLECI`, etc.) are **not** consulted. `CI=true` is set by every modern runner, and per-platform lists rot over time.

Auto-detection is strictly a **presentation** decision. No command behavior, validation, exit code, or other side effect may vary based on the detected mode — this is the invariant that keeps `--output`'s auto-switch from becoming the `CI=true`-changes-semantics footgun (create-react-app's canonical cautionary tale). See [[#S2-14 Output Mode Auto-Detection|S2-14]].

#### Per-command JSON discipline

The shape of stdout in `json` mode differs by command to match the dominant idiom for each command type:

- **`validate`** (validator idiom, matching `ruff` / `biome` / `eslint`): emits a single JSON document on stdout on **both pass and fail** paths. An empty diagnostics array signals a clean pass; a populated array signals failure. The exit code redundantly signals the outcome (0 pass, 1 fail). stderr stays quiet.
- **`package`, `yank`, `new`** (data-tool idiom, matching `kubectl` / `gh` / `aws`): on success, emit a single JSON document on stdout describing what was written (paths, hashes, index changes). On failure, stdout stays empty; a human-readable error line is written to stderr and the exit code is non-zero. `OUT=$(influxdb3-plugin package --output json ...)` under `set -e` behaves correctly — `$OUT` is either a complete JSON document or the pipeline has already failed.

**Invariant across all commands in `json` mode:** stdout is either empty (data-tool failure path) or a single well-formed JSON document — never partial writes, concatenated documents, or mixed human text on stdout. This is what makes `| jq` reliable. See [[#S2-15 Per-Command JSON Output Discipline|S2-15]].

The top-level `--version` flag is exempt from `--output` — it always prints a single line of text regardless of mode.

#### Schema stability

The JSON output shape is part of the `influxdb3-plugin-cli` crate's public semver contract (see [[#Stability & Compatibility]]). Adding new fields is a minor bump; renaming, removing, repurposing, or narrowing the type of an existing field is a major bump. Consumers must ignore unknown fields.

The output does **not** embed an `output_schema_version` field. Consumers pin via the crate's published version, matching the dominant modern CLI convention for invocation-result output (`ruff`, `biome`, `uv`, `gh`, `rustc --error-format=json`, `cargo --message-format json`). Embedded schema versions are standard for persistent document formats that outlive their invocation (OCI manifests, Kubernetes API types, Terraform plan output) but not for invocation-result CLI output — and this output is an invocation result, not a persistent document. The persistent documents this tool manipulates (manifests, indexes, artifacts) already carry their own schema versions per [[#Spec 1: Source & Distribution Contracts]]'s [[#Schema Versioning Strategy]], so consumers always have a version to pin against at the layer that matters.

See [[#S2-16 JSON Output Schema Stability|S2-16]].

### Release Engineering

`INFLUXDB3_PLUGIN_SDK_KNOWN_LATEST_DB`, the build-time constant that backs the default for `new --database-version`, is supplied to the SDK as an env var (name prefixed per [[#S2-9 Env-Var Namespace|S2-9]]) and read by `influxdb3-plugin-sdk/build.rs`:

1. `build.rs` reads the env var. When set, it validates that `>={value}` parses as a `semver::VersionReq` and panics the build on invalid input — no silent bake-in of an unparseable value that would later reach scaffolded manifests.
2. `build.rs` emits `cargo::rustc-env=INFLUXDB3_PLUGIN_SDK_KNOWN_LATEST_DB=<value>` and `cargo::rerun-if-env-changed=INFLUXDB3_PLUGIN_SDK_KNOWN_LATEST_DB`; `scaffold::DEFAULT_DATABASE_VERSION` then reads the value via `concat!(">=", env!("INFLUXDB3_PLUGIN_SDK_KNOWN_LATEST_DB"))`.
3. When the env var is unset, `build.rs` falls back to `3.0.0`, emits a `cargo::warning` announcing the fallback, and sets the `sdk_known_db_is_fallback` cfg flag so fallback-sensitive tests can gate on the build-layer truth rather than guessing from process env at test time.

Release CI is responsible for supplying a real value to the build:

1. Pre-build step fetches the latest `influxdb3` version from GitHub Releases on `influxdata/influxdb` and passes it to the build via `INFLUXDB3_PLUGIN_SDK_KNOWN_LATEST_DB=<value>`.
2. Release fails if the lookup fails — no silent fallback to a stale value.

The "no silent fallback" rule is a release-layer contract, not a build-script contract: the build script accepts the dev-build fallback precisely so `cargo build` works offline without any CI scaffolding. CI's responsibility is to ensure that every shipped binary carries a real, fetched value.

There is no scheduled re-release. Staleness is bounded by SDK release cadence; authors who need a different default use `--database-version <range>` explicitly.

### Distribution

This section describes how the SDK binary reaches plugin authors and CI/CD agents. Release automation tooling (raw GitHub Actions matrix, `cargo-dist`, or equivalent) is implementation detail and not part of this spec.

#### Platform matrix (v1)

The SDK ships pre-built binaries for four targets, mirroring `influxdb_pro`'s release matrix exactly so the SDK's release pipeline reuses the org's existing cross-builder image:

| OS | Architecture | Rust target triple |
|---|---|---|
| Linux | x86_64 | `x86_64-unknown-linux-gnu` |
| Linux | aarch64 (Graviton, ARM CI) | `aarch64-unknown-linux-gnu` |
| macOS | aarch64 (Apple Silicon) | `aarch64-apple-darwin` |
| Windows | x86_64 | `x86_64-pc-windows-gnu` |

Linux binaries dynamically link glibc; macOS and Windows binaries have no non-OS runtime dependencies. Alpine Linux, Intel macOS, and Windows-on-ARM are out of scope for pre-built v1 binaries; users on those platforms install via `cargo install --git` (per [[#S2-20 v1 Distribution Channel|S2-20]]) or rebuild from source. See [[#S2-19 Platform Matrix|S2-19]].

#### Canonical distribution (v1)

**GitHub Releases is the sole supported distribution channel for v1.** Each tagged release publishes:

- One `influxdb3-plugin-<target-triple>.tar.gz` per target from [[#S2-19 Platform Matrix|S2-19]]. Tarball is used for all targets including Windows, mirroring `influxdb_pro`'s release artifact convention.
- A `SHA256SUMS` file covering every asset. Consumers verify integrity via SHA-256 in v1; cryptographic signing is planned post-v1 (see [[#TODOs]]).
- Release notes sourced from the changelog.

See [[#S2-20 v1 Distribution Channel|S2-20]].

**CI/CD agent consumption patterns.** Two widely-used shapes both work against GitHub Releases alone:

*Direct `curl` + checksum verification* — portable across every CI platform (GitHub Actions, GitLab CI, Jenkins, CircleCI, Buildkite):

```bash
VERSION=v1.0.0
TARGET=x86_64-unknown-linux-gnu
curl -sSfL -o sdk.tar.gz \
  "https://github.com/influxdata/influxdb3-plugin-sdk/releases/download/${VERSION}/influxdb3-plugin-${TARGET}.tar.gz"
sha256sum --check <<< "${EXPECTED_SHA256}  sdk.tar.gz"
tar xzf sdk.tar.gz && sudo mv influxdb3-plugin /usr/local/bin/
```

*GitHub Actions ergonomic wrapper* via `taiki-e/install-action`, which resolves the release from GitHub Releases:

```yaml
- uses: taiki-e/install-action@v2
  with:
    tool: influxdb3-plugin
```

Registration with `taiki-e/install-action` is a one-time upstream PR and is the canonical consumption path for Rust CLIs in GitHub Actions — `ruff`, `uv`, `cargo-nextest`, `dprint`, and hundreds of other tools are consumed this way. No additional distribution channel beyond GitHub Releases is required to enable this pattern.

#### Post-v1 channels

The following additional channels are planned but not part of the v1 distribution commitment. Each is additive — they layer on top of the GitHub Releases canonical path, not in place of it.

- **Homebrew tap** — `brew install influxdata/tap/influxdb3-plugin` for macOS and Linux brew users.
- **winget** — `winget install InfluxData.InfluxDB3Plugin` for Windows users.
- **`cargo install influxdb3-plugin-cli`** via crates.io — source install for Rust users without internal-repo access. Gated on completion of security and legal review of the crate's publication. The `cargo install --git` form is already a v1 channel for users with internal-repo access (see [[#S2-20 v1 Distribution Channel|S2-20]]).
- **Distroless container image** — for CI workflows that prefer containerized tooling.
- **`.deb` / `.rpm` packages** — for OS-level package management.
- **Signed installer script** (`curl | sh` with checksum + Sigstore-verified pinned download) — lands with the broader signing/provenance work.
- **Self-update capability** (`influxdb3-plugin self-update`) — must be disabled automatically when `CI=true` to preserve reproducibility.

See [[#TODOs]].

### Packaging

The `package` command takes a plugin directory plus an input index and produces a derived index containing the new entry alongside a versioned artifact. See [[#Commands]] for the command-line surface.

**Inputs:**
- An authored plugin directory containing `manifest.toml` and `__init__.py`.
- An input `index.json` passed via `--index`. Read-only; never modified (S2-11).

**Outputs (both in `--out`):**
- `<out>/index.json` — the input index with the new plugin-version appended. `artifacts_url` and all existing entries are preserved verbatim.
- `<out>/<name>-<version>.tar.gz` — the new artifact.

Steps:
1. Validate the manifest (see [[#Validation]]). All validation errors are collected and reported together rather than failing on the first.
2. Reject if `(name, version)` already exists in the input index (S2-2).
3. Bundle the plugin directory into a gzipped tar archive, excluding files not needed for execution.
4. Compute SHA-256 of the archive → `hash`.
5. Write the derived index (input + new entry with `hash` and `published_at`) to `<out>/index.json`.
6. Write the archive to `<out>/<name>-<version>.tar.gz`.

The tool is the sole writer of the index (S1-2). Archive outputs are byte-deterministic given identical inputs; derived-index outputs are byte-deterministic given identical inputs plus the publication timestamp for any newly added entry (S2-3). The tool has no network awareness — moving outputs to a registry is the author's or CI's responsibility (see [[#Publish Model]]).

### Validation

The SDK's validation step — invoked directly by `validate` and as the first step of `package` — performs the checks below. All errors are collected and reported together rather than failing on the first; authors fix everything in one pass.

**Layered error collection.** Validation errors are collected at two layers: the `influxdb3-plugin-schemas` crate collects structural and per-field errors (name regex, SemVer, URL scheme, PEP 508, description length, hash format, duplicate `(name, version)`, etc. — the rules listed below that depend only on a single parsed document) during its two-phase parse (see Spec 2 § Error model). The `influxdb3-plugin-sdk` crate's `validate` entry point adapts those schema-level `ReportedError`s losslessly into the SDK's single diagnostics container (`ValidationError` / `ValidationReport`, or a direct rename thereof) and uses that same container for cross-file errors (required-file presence, code/manifest cross-reference, index-relative uniqueness when `--index` is supplied). When structural parsing succeeds, later checks append to that same report; when structural parsing fails, those structural diagnostics are returned in the same SDK-side shape and downstream cross-file checks are skipped because they depend on a parsed manifest. The CLI therefore sees one diagnostics shape at the SDK boundary rather than branching between separate structural-vs-cross-file error containers.

**Structural (manifest + required files):**
- Manifest is well-formed TOML and conforms to the declared `manifest_schema_version`.
- Required files exist at the plugin-dir root: `manifest.toml` is always required. The Python entry point is detected as either `__init__.py` (multi-file plugin) or the sole regular `.py` file at the top level (single-file plugin). Symlinks are excluded from detection. Two new structural error cases apply:
  - **No entry point:** no regular `.py` files exist at the top level.
  - **Ambiguous entry point:** multiple `.py` files exist at the top level but none is `__init__.py`.
- `plugin.name` matches the rule `[a-zA-Z][a-zA-Z0-9_-]*` (1–64 ASCII chars, starting with a letter; case-preserving) and is not a Windows reserved device name (`con`/`prn`/`aux`/`nul`/`com0-9`/`lpt0-9`, case-insensitive).
- `plugin.version` is SemVer 2.0.0 compliant.
- `plugin.description` is non-empty, contains no line breaks (CR or LF), and ≤ 200 characters.
- `plugin.triggers` is a non-empty array whose entries are drawn from the closed set `{process_writes, process_scheduled_call, process_request}`.
- `plugin.homepage`, `plugin.repository`, `plugin.documentation` — when present — parse as valid URLs with `http://` or `https://` scheme.
- `dependencies.database_version` parses as a valid SemVer range.
- `dependencies.python` entries are PEP 508-parseable.

**Code / manifest cross-reference**:
- The entry point file parses as valid Python 3 syntax.
- For each identifier in `plugin.triggers`, a top-level synchronous `def <name>(...):` exists in the entry point file. Shifts "declared but not implemented" from trigger-execute-time on the db to package-time on the author's machine. Two explicit non-matches, both by design:
  - `async def <name>` is rejected. The runtime invokes trigger functions synchronously via PyO3's `call1()` and cannot await coroutines; matching the runtime's lenient `getattr` tolerance would falsely certify `async def` as valid and defer the failure to production.
  - Indirect definitions — re-exports (`from .handler import process_writes`), module-level assignments (`process_writes = some_callable`), class-based callables — are not recognized. Only plain top-level `def` appears in current production plugins and bundled test fixtures. Authors adopting an indirect pattern will need a future opt-out; v1 provides none.

**Index-relative** (only when an input index is available — always for `package`; optional `--index` for `validate`):
- `(name, version)` does not already exist in the index (immutable publication, S2-2).
- The canonical form of `name` (`lowercase(name).replace('-', '_')`) does not collide with any existing entry under a different spelling — across any version. Two canonically-equivalent spellings in one registry are rejected as a distinct failure mode from `(name, version)` immutability; the actionable fix is to rename, not to bump or yank. See S2-2.

**Explicitly deferred from v1:**

- **Python import ↔ declared dep consistency.** Requires an import-name → distribution-name map (e.g., `google.cloud` ↔ `google-cloud-storage`) the SDK cannot derive locally, plus version-specific stdlib module sets that shift across Python versions. Strict enforcement produces false positives; warning mode needs a warning channel (v1 has no non-error diagnostic tier — exit codes are GNU baseline `0`/`1`/`2`, and stderr has no documented warning stream). Revisit when the SDK has optional network-aware features and a warning UX.
- **Version-incremented check.** Strict "monotonically increasing" breaks legitimate multi-release-line patterns (`1.0.0` → `2.0.0` → `1.0.1` patch). The useful form is a *warning* when the new version is not the highest in the index. Requires a warning channel the v1 SDK doesn't have. Revisit with warning UX.
- **Stale-version-copy detection.** An author who copies a plugin from elsewhere and publishes without bumping the manifest's `version` can unwittingly publish a lower version than the current release line (e.g., `0.1.0` when the index already contains `1.2.0`). S2-2 catches exact-version conflicts but not below-current-line versions. Detecting this requires either strict version-increment enforcement (rejected above) or a warning channel. Rare in practice; v1 accepts the gap and relies on author review of the manifest before publish.

### Reproducibility

S2-3 requires byte-identical outputs given identical inputs. This section defines what counts as an input and the canonicalization rules that make the guarantee hold. For derived indexes that add a new entry, publication time is an input: production packaging uses current UTC, while deterministic tests and automation can supply a fixed timestamp through the SDK library path.

**Guarantee scope.** The same SDK version run on different machines with the same package inputs produces identical `<name>-<version>.tar.gz` bytes. The same SDK version run with the same package inputs, same input index, and same publication timestamp for any newly added entry produces identical `index.json` bytes. Different SDK versions may produce different output bytes — the SDK version is an implicit input. Cross-SDK-version determinism is explicitly not promised.

**Archive canonicalization (tar.gz):**
- **Tar format**: `ustar`. Explicitly selected — Rust's `tar` crate defaults to GNU, which is not byte-identical to ustar for the same logical content.
- **Entry ordering**: all entries sorted by path in lexicographic UTF-8 byte order before emission.
- **mtime on every entry**: `0` (Unix epoch). Filesystem mtimes are never encoded.
- **UID / GID**: `0` / `0`. Owner and group name fields: empty strings.
- **File mode**: files → `0644`; files with the executable bit set on disk → `0755`. The archive contains regular-file entries only — no directory records, no symlinks, no device nodes, no FIFOs. Directory structure is implicit in file paths and reconstructed by the extractor with its default permissions. This restriction avoids the traversal-order ambiguity empty directories would introduce for reproducibility.
- **PAX extended headers**: none. Any metadata that would require pax headers (ACLs, file caps, sparse-file metadata, extended attributes) is either stripped or the archive is rejected at package time.

**Gzip wrapper canonicalization:**
- **Gzip header timestamp**: `0`.
- **Original filename header**: omitted.
- **Compression**: `flate2` crate with the `miniz_oxide` backend, fixed compression level `6`. The backend is pinned because OS-provided zlib implementations produce byte-different output at the same level.

**Derived index canonicalization (`index.json`):**
- **Field ordering**: matches struct declaration order of the index schema type in `influxdb3-plugin-schemas`, which in turn matches the field ordering shown in Spec 1's Index Schema example (`name`, `version`, `published_at`, `description`, `triggers`, URL fields, `dependencies`, `hash`, `yanked` when present). Alphabetical sort is not used — the schema-specified order is more readable in diffs.
- **`plugins[]` ordering**: sorted by `name` ascending, then `version` ascending per SemVer 2.0.0 precedence. Yanked status does not affect ordering.
- **Indentation**: 2 spaces, pretty-printed. The index is routinely reviewed in PRs on the registry backend (Git-hosted registries especially) — human-diffability matters.
- **Trailing newline**: present at end of file.
- **Unicode normalization**: NFC. No practical effect in v1 since plugin names are ASCII-only per the name rule; specified for forward-compat.

**What counts as an input** (contributes to output bytes):
- Plugin directory contents: file paths, file bytes, filesystem executable bit.
- The file-exclusion artifact (when present). The specific mechanism is deferred — candidates are a `.pluginignore` pattern file alongside `manifest.toml`, or an explicit include list in `manifest.toml` (e.g., `plugin.files = [...]`). Whichever mechanism lands, its contents are an input to the archive bytes. See [[#TODOs]].
- Input `index.json` contents (when `--index` is provided).
- Publication time for any newly added index entry. Production packaging uses current UTC. Tests and automation that need byte-identical derived indexes must inject a fixed timestamp through the SDK library path.
- SDK version (implicit).

**What does NOT count as an input** (must not affect output bytes):
- Machine hostname, username, UID, GID.
- Wall-clock time at build must not affect archive bytes. It affects derived-index bytes only when production packaging uses current UTC as the default `published_at` for a newly added entry.
- Filesystem timestamps (`mtime`, `atime`, `ctime`).
- Absolute path to the plugin directory.
- Environment variables.
- Terminal encoding, locale, `LC_ALL`.
- Parent directory layout outside the plugin directory.

**Rationale: no `SOURCE_DATE_EPOCH` handling.** The reproducible-builds.org convention is for tools to honor `SOURCE_DATE_EPOCH` as the source of timestamps in output. The SDK deliberately does not use that variable. Tar entry mtimes and gzip header timestamps are pinned to zero unconditionally, which keeps archive bytes reproducible regardless of environment. `published_at` is not a build timestamp; it is explicit registry metadata for a published plugin version. Production packaging defaults it to current UTC, and deterministic callers that need identical derived-index bytes provide a fixed value. Honoring `SOURCE_DATE_EPOCH` would blur build reproducibility with publication metadata and widen the input set to include ambient environment configuration.

**Testing.** The reproducibility invariant is verified by a test matrix. Archive determinism tests perturb each non-input — machine hostname, username, UID, GID, wall-clock time, filesystem timestamps, absolute path to the plugin directory, environment variables, terminal encoding and locale, parent-directory layout — between two invocations with otherwise-identical logical inputs and assert byte-identical archive output. Derived-index exact-byte tests use fixed publication timestamps; production-clock behavior is covered separately with before/after UTC bounds and strict format assertions. A single-dir, two-call purity test is not sufficient on its own: it exercises only in-process determinism and would pass even if one of the listed non-inputs leaked into output bytes. Testing occurs within a single SDK version; cross-SDK-version reproducibility is not promised and not tested (see the "Guarantee scope" paragraph above).

### Publish Model

Publishing is separate from packaging. The packaging tool produces local files only — it has no network awareness, no authentication, and no knowledge of where artifacts will be hosted. Uploading the artifact and updated index to the registry backend (S3 upload, Git commit, HTTP server copy, GitHub Release asset upload, etc.) is a separate step handled by the author, a script, or CI.

This keeps the tool portable and ensures it works in fully offline environments.

**Concurrent publish race.** The immutable-input model permits two `package` runs against the same input index to produce two derived indexes for different plugin-versions in parallel, neither aware of the other. Conflict detection lives at upload time via the registry backend's concurrency control (Git push fast-forward; S3 conditional PUT; object-store `If-Match`). The SDK enforces no cross-run coordination. Handling is an author/CI responsibility; pipeline mechanics are out of Spec 2 scope (see [[#TODOs]]).

**CI retry and idempotency.** `package` has no idempotent-replay flag in v1; it rejects duplicates in the input index per [[#S2-2 Immutability Check|S2-2]]. The retry scenario this leaves open — a prior CI run's upload half-succeeded, the registry state now reflects the new entry, and the retry's `package` trips S2-2 against the updated index — is handled at the CI layer rather than inside the SDK:

- **Concurrency groups.** Serialize publish workflows against a given registry (GitHub Actions `concurrency: group: plugin-publish-<registry>, cancel-in-progress: false`; GitLab `resource_group:`) so race-with-self cannot produce partial state.
- **Upload-side idempotency.** The upload step (outside the SDK) uses conditional-write primitives appropriate to the backend: `aws s3api put-object --if-none-match '*'` for object stores, `git push --force-with-lease=refs/heads/main:refs/heads/main` for Git-hosted indexes, `gh release upload --clobber=false` for GitHub Release assets. When target state already matches intent, the upload is a no-op and the job continues.
- **Preflight check.** The CI workflow fetches the current registry index and checks whether `(name, version)` already exists before running `package`. If it does and this run isn't an intentional reissue, the job exits success early without invoking `package`.

This keeps the SDK network-unaware per [[#S2-1 Tool Is Network-Unaware|S2-1]] and aligns with dominant packaging-tool convention: `cargo package`, `npm pack`, `helm package`, and `goreleaser build` all produce local artifacts without registry awareness; retry and idempotency are solved one layer up, in the publish/upload tool or the CI workflow itself. A `--allow-exists` opt-in flag for local idempotent replay (analogous to `twine upload --skip-existing` / `dotnet nuget push --skip-duplicate`) is an additive v1.x change if real CI demand materializes — see [[#TODOs]].

**Preview pattern.** Output is byte-deterministic per [[#S2-3 Deterministic Regeneration|S2-3]], so running `package` or `yank` with `--out` pointed at a scratch directory is equivalent to a `--dry-run` — the scratch contents are exactly what a real run would produce and can be inspected for hash, derived-index entry, or yank-flag diff before being promoted or discarded. `validate` covers the validation-layer preview; the scratch-dir pattern covers the artifact- and index-layer previews. No dedicated `--dry-run` flag ships in v1 (see [[#TODOs]]).

### Author Workflows

All seven workflows from [[#Author workflow]] are supported under the Spec 1 identity model:

| Workflow | How it works |
|---|---|
| Local dev without git (**DECIDED**) | Local install path; no registry, no packaging. |
| Local dev with git | Clone upstream, iterate with local install path or a localhost registry (`file://` or `http://localhost:8080`). |
| Remote repo / CI/CD | CI runs the packaging tool on merge and uploads artifact + regenerated index to the configured backend. |
| Cross-platform local dev | Schema is platform-neutral (TOML/JSON, URLs); platform-specific concerns live in the extraction codepath. |
| Contributing to someone else's plugin | Fork produces a distinct `(index_url, name, version)` identity; no risk of silent substitution against upstream. |
| Migrating an existing plugin | Author writes a v1 manifest and runs the packaging tool. Destination registry follows its own `artifacts_url`. |
| Yanking | Toggle `yanked: true` on the existing entry; artifact remains fetchable. |

### Requirements

#### S2-1: Tool Is Network-Unaware

The packaging tool writes artifacts and the index to a local output directory only. Transport is out of scope for the tool. Correspondingly, no SDK command accepts credential flags (API tokens, AWS keys, git SSH material, etc.); credentials are the upload step's concern and never enter the SDK process. Keeps the SDK fully auditable as an offline-safe tool.

**No implicit filesystem discovery.** The tool reads from the filesystem only the paths explicitly named in its arguments: the plugin directory (`[plugin-dir]` positional), the file passed via `--index`, and the template source paths internal to `new`. It does not auto-discover configuration files, index files, or other state from conventional paths — no lookups in ancestor directories, the user's home directory, `$XDG_CONFIG_HOME`, or anywhere else the user did not explicitly name on the command line. Omitting an optional argument (e.g., `validate` without `--index`) means the corresponding operation is not performed — not that a default location is consulted.

#### S2-2: Immutability Check

The packaging tool must reject packaging under two distinct failure modes, keyed on [[#Plugin Identity Model|canonical name]] (`lowercase(name).replace('-', '_')`):

- **Exact duplicate.** `(name, version)` already exists in the target index with the same spelling and the same version. Actionable fix: increment `plugin.version` in `manifest.toml` or run `yank` to retract the prior entry. The error output must list every existing version of the plugin in the input index so the author can pick a higher version without re-querying.
- **Canonical collision.** An existing entry shares the new entry's canonical name but spells it differently (e.g. `foo-bar` vs. `foo_bar`, `MyPlugin` vs. `myplugin`). Fires regardless of version — two canonically-equivalent spellings in one registry are always rejected. Actionable fix: rename to one of the existing spellings or choose a distinct name. The error output must enumerate the existing `(spelling, version)` pairs under that canonical form so the author can see which spelling the registry already uses.

Both failure modes surface as distinct error variants in the SDK (and in `--output json`) so downstream tooling can render fix-appropriate guidance without parsing error text. Without this context, "already exists" errors leave first-time authors stuck.

Problems: [[#No published-version lifecycle controls]]

#### S2-3: Deterministic Regeneration

Given identical inputs (plugin sources and prior index state), the tool must produce byte-identical outputs modulo content that inherently varies.

#### S2-4: Async Entry Point

The SDK's top-level embeddable config type (`PluginConfig` or equivalent) exposes `pub async fn run(self) -> anyhow::Result<()>`. Errors flow through `Result`; the function never calls `std::process::exit`. The async signature lets the type embed cleanly inside a host binary's existing tokio runtime without spawning a nested one.

#### S2-5: Self-Declared Version

The SDK's top-level embeddable config type carries a `#[command(version = …)]` attribute sourced from its own crate metadata (e.g., `env!("CARGO_PKG_VERSION")`) so that `<host> plugin --version` reports the SDK's version independently of the host binary's version. Without this, embedded subcommands cannot surface their version at all — clap does not inherit `--version` across subcommand boundaries.

#### S2-6: Clap Version Range Policy

The SDK pins its clap version range to be compatible with `influxdb_pro`'s *resolved* clap version, not its nominal workspace pin. The true floor is determined by `influxdb_pro`'s transitive constraints (currently `^4.5.47`, set by `clap_builder`); the SDK must not pin below this. The SDK's pin also must not force an undesired downgrade of `influxdb_pro`'s clap when the two are unified in a future phase-2 build. Clap major-version upgrades are coordinated across repos.

#### S2-7: No Direct Process Exit

The SDK never calls `std::process::exit`. All exit control stays with the host binary, which owns exit-code semantics, runtime shutdown ordering, and cleanup of global state (allocator, TLS, panic hook, tracing subscriber). Enforced in the SDK crate via a clippy `disallowed-methods` lint on `std::process::exit`.

#### S2-8: No Global-State Installation

The SDK's library surface installs no process-global state. This includes (non-exhaustively):

- `#[global_allocator]`
- TLS/crypto providers (`rustls`, `native-tls`, `openssl`, `aws-lc-rs`)
- Nested tokio runtimes
- Tracing subscribers and log backends (`tracing::subscriber::set_global_default`, `tracing_subscriber::fmt::init`, `env_logger::init`, `log::set_boxed_logger`, `console_subscriber::init`)
- Panic hooks (`std::panic::set_hook`)
- Signal handlers (`ctrlc`, `signal-hook`)
- OpenTelemetry providers (`opentelemetry::global::set_tracer_provider`, `set_meter_provider`, `set_logger_provider`)

These are host-owned concerns. Under embedded invocation, the host has already configured them before `PluginConfig::run()` is called; the SDK library would clobber or race that setup.

**Scope: library surface only.** The SDK's standalone binary entry point (`influxdb3-plugin-cli`'s `main.rs`) is free to install any of the above for standalone-mode use — without them, standalone invocation would produce no logs, no backtraces, and no Ctrl-C handling. The separation ensures standalone invocation is usable and embedded invocation leaves the host in full control.

Any library-side exception requires a feature flag disabled by default. Where mechanically enforceable, a clippy `disallowed-methods` lint in the SDK crates names the above functions.

#### S2-9: Env-Var Namespace

All clap `env = "…"` bindings in the SDK's CLI surface use the `INFLUXDB3_PLUGIN_*` prefix to avoid collisions with the host's existing `INFLUXDB3_*` env-var surface.

#### S2-10: Schema Types from Shared Crate

Manifest, IndexEntry, and related parse-time types used by the SDK live in the `influxdb3-plugin-schemas` crate and are not redefined locally. Any future db-runtime code that parses these types (Specs 3 and 4) depends on the same versioned crate. Schema evolution follows the rules established in [[#Schema Versioning Strategy]].

#### S2-11: Immutable Input

The `package` and `yank` commands must not modify the file passed to `--index`. The input index is read-only; the derived index is written to `--out`. Preserving the input enables deterministic regeneration (S2-3), safe CI flows, and reviewable diffs.

Problems: [[#No published-version lifecycle controls]], [[#No artifact integrity verification]]

#### S2-12: Input/Output Separation

The directory passed to `--out` must not resolve to the directory containing the file passed to `--index`. The tool must reject configurations that would permit self-overwrite. Safety rail for S2-11.

#### S2-13: Output Mode Flag and Defaults

Every SDK command must accept `--output <mode>` with the v1 value set `{human, json}`. Unsupported values fail at argument-parse time with an error naming the accepted set. `human` targets interactive CLI use; `json` produces a machine-readable document on stdout per [[#S2-15 Per-Command JSON Output Discipline|S2-15]]. Additional modes (e.g., `sarif`, `yaml`, `github`, `gitlab`) are reserved for additive future changes through the same enum flag and must not be accepted in v1.

The top-level `--version` flag is exempt and always prints plain text.

Problems: none directly — this is a behavior contract that makes SDK output consumable by CI without breaking the interactive experience.

#### S2-14: Output Mode Auto-Detection

When `--output` is omitted, the SDK must select `json` when `!isatty(stdout)` OR when the `CI` environment variable is set to a truthy value (`true`, `1`); otherwise `human`. Explicit `--output <mode>` always wins. Detection must use Rust stdlib `IsTerminal` and must not consult platform-specific CI env vars (`GITHUB_ACTIONS`, `GITLAB_CI`, `JENKINS_URL`, `BUILDKITE`, `CIRCLECI`, etc.).

Auto-detection is a **presentation-only** decision: no command behavior, validation logic, exit code, file-write outcome, or other side effect may vary based on the detected mode. This is the guarantee that keeps auto-detection from becoming a source of environment-dependent behavior drift (the `CI=true`-changes-semantics anti-pattern).

Problems: none directly — this is an implementation contract.

#### S2-15: Per-Command JSON Output Discipline

In `--output json` mode, stdout obeys the following per-command contract:

- **`validate`** must emit a single well-formed JSON document on stdout on both pass and fail paths. Diagnostics live in a top-level array; the exit code redundantly signals pass (0) or fail (1). stderr stays quiet — the stdout document is the only signal on both paths.
- **`package`, `yank`, `new`** must emit a single well-formed JSON document on stdout only on success. On failure, stdout must stay empty; a single-line human-readable error is written to stderr and the exit code is non-zero. Multi-line diagnostic blocks on stderr are forbidden in `--output json` mode so that `tool 2>&1 | head -1` and similar CI captures remain well-defined.

Across all commands in `json` mode, stdout is either empty (failure path for data-tool commands) or a single well-formed JSON document — never partial writes, concatenated documents, or mixed human text. stderr is bounded: quiet for `validate`; exactly one meaningful line for the data-tool trio's failure path.

**Clap parse errors are bound by the same single-line rule.** Clap's default multi-line diagnostic (`error: …`, `[possible values: …]`, blank line, `For more information, try '--help'.` footer) is collapsed to one meaningful line on stderr in `--output json` mode; UX affordances printed alongside the clap diagnostic — for example the `new list` pointer surfaced on unknown-template errors — are suppressed in JSON mode, since a second line would violate the contract. Human mode preserves clap's full rendering, including such affordances.

**Human-mode stderr is NOT similarly bounded.** In `--output human` mode, a data-tool command's failure path MAY emit a multi-line diagnostic block on stderr — this is where `package`'s validation-failure rendering lives per the Packaging "authors fix everything in one pass" intent. The single-line restriction above applies only to `--output json`.

Problems: none directly — this is a behavior contract that makes `| jq` and `OUT=$(...)` patterns safe under `set -e`.

#### S2-16: JSON Output Schema Stability

The JSON output shape of every CLI command in `--output json` mode is part of the `influxdb3-plugin-cli` crate's public semver contract. Adding fields is a minor-version change; renaming, removing, repurposing, or narrowing the type of an existing field is a major-version change. Consumers must ignore unknown fields.

**Enumerated string values** appearing in JSON output are part of the schema contract on the same terms as field names. This includes `variant` discriminators in error documents, `outcome` markers in state-transition responses (e.g., `yank`'s `transitioned` / `already_in_desired_state`), and any closed-set string-valued field. Adding a new enumerated value (e.g., introducing a new error variant such as `PathTooLong`) is a minor-version change; renaming, removing, or reassigning an existing enumerated value to a different failure class is a major-version change. Consumers may safely pattern-match on these values with the expectation they will not silently change within a major version. This matches the stability posture of `rustc --error-format=json`'s diagnostic codes and `ruff`'s rule identifiers.

No `output_schema_version` field is embedded in the output. Consumers pin via the crate's published version, matching the convention established by modern invocation-result CLI output (`ruff`, `biome`, `uv`, `gh`, `rustc --error-format=json`, `cargo --message-format json`). Persistent documents that the tool emits (manifests, index entries, artifacts) carry their own schema versions per [[#Schema Versioning Strategy]].

Problems: none directly — this is an implementation contract that keeps CI consumers of SDK output from breaking silently on SDK upgrades.

#### S2-17: Stream Separation, Color Handling, and Non-Interactive Posture

Every SDK command, in both `human` and `json` output modes, obeys a strict stdout/stderr contract.

**Stream separation.** stdout carries the command's primary result only:

- In `human` mode: the human-readable rendering of what the command produced (e.g., the "packaged X → <path>, hash <sha>" confirmation for `package`, the diagnostics list for `validate`, the scaffold confirmation for `new`, the yank-state transition for `yank`).
- In `json` mode: the JSON document per [[#S2-15 Per-Command JSON Output Discipline|S2-15]].

stderr carries everything else — human error lines on failure, warnings, and any future verbose/debug output when those flags are added post-v1. `tool | less`, `tool | jq`, `tool > file`, and `OUT=$(tool ...)` must all work without capturing diagnostic text.

**No interactive I/O in v1.** The SDK emits no spinners, progress bars, banners, or prompts in either mode. A command that lacks required input fails at argument-parse time with an error naming the missing flag; it must not block on stdin and must not attempt to recover by prompting. Adding any of these is a v1-scope change and requires a separate follow-on requirement. This generalizes the non-interactive posture the `new` command already documents ("errors and asks for an explicit `--name`").

**Color handling.** Color may be rendered on a stream when all of the following hold:

| Condition | Effect |
|---|---|
| `NO_COLOR` set to any non-empty value | Color disabled on all streams (per [no-color.org](https://no-color.org/)). Overrides everything below. |
| `TERM=dumb` | Color disabled on all streams. |
| `FORCE_COLOR` set to any non-empty value | Color enabled on all streams regardless of TTY detection. |
| `isatty(<stream>)` is true and none of the above disable rules fired | Color enabled on that stream. |
| `isatty(<stream>)` is false and no force rule fired | Color disabled on that stream. |

**Absolute rule, regardless of any of the above:** color is never emitted on stdout in `json` mode. JSON on stdout must be byte-stable and parseable; ANSI escapes break `jq` and every other consumer.

**Why this is a single requirement:** stream separation, the no-prompt posture, and color rules together form the contract that makes both interactive `tool | less` and unattended CI capture work. Splitting them would fragment the contract without adding clarity.

Problems: none directly — this is the "stdout is data; stderr is everything else" rule, the single most-cited CLI-design discipline and the foundation that makes piping, CI capture, and auto-detection ([[#S2-14 Output Mode Auto-Detection|S2-14]]) reliable.

#### S2-18: Exit Code Set

Every SDK command returns exit codes from the GNU baseline in v1:

- **`0`** — success. Includes any idempotent-no-op success paths added later through flag opt-in.
- **`1`** — runtime failure. The operation the user requested did not complete successfully. Covers validation failure, file I/O error, immutability conflict (e.g., `(name, version)` already exists per [[#S2-2 Immutability Check|S2-2]]), index parse error, internal invariant-check failure, and any other non-usage failure mode.
- **`2`** — usage error. The command was invoked incorrectly. Clap emits this automatically on parse-time errors; command code must also return `2` for arg-shape errors caught after parse. Covers (non-exhaustively):
  - Unrecognized flags.
  - Missing required arguments.
  - Invalid values for closed-set flags (`--output`, `--name`, etc.).
  - **Plugin-name rule violation on an explicit `--name` value** (e.g., `--name "Bad Name"` — `PluginName::from_str` rejected the input). Includes both character-rule failures and Windows-reserved-name rejections.
  - **Flag/template mismatches** (e.g., `--artifacts-url` on a plugin template, `--database-version` or `--name` on the `registry` template). A flag that is only meaningful for a specific template is a usage error when passed to any other template.
  - **Unparseable `--database-version` value.** `new <plugin-template> --database-version <bad>` exits 2 when `<bad>` is not a valid `semver::VersionReq`. The CLI pre-validates before the scaffold writes anything; no manifest is produced.
  - **Unsupported `--artifacts-url` scheme or malformed URL.** `new index --artifacts-url <bad>` exits 2 when `<bad>` is not a parseable URL or uses a scheme outside `{https, http, file}` (see [[#Index Schema]] for the allowed set). The CLI pre-validates before the scaffold writes anything; no `index.json` is produced.
  - **Malformed positional arguments** (e.g., `yank name:version` where the positional must be `<name>@<version>`). SemVer and plugin-name parse failures inside a positional argument also qualify.
  - Input/output directory aliasing caught by [[#S2-12 Input/Output Separation|S2-12]].
  - Any other argument-parse or early-validation failure where the condition is under caller control rather than environmental.

  Derived values that are invalid while the caller-supplied input was well-formed may remain exit `1` — e.g., `new process_writes "Bad Path"` where the path argument is valid but the derived plugin name is not; the runtime-failure exit code is consistent with "the input you gave me was legal, the system couldn't complete the operation." Callers can disambiguate by passing `--name` explicitly, which routes through the exit-`2` path above.

Exit codes `3` through `255` are reserved for additive future semantic codes. Any future code must be documented, numerically higher than `2`, tied to a specific CI-branching use case that cannot be served by a flag-gated exit-`0` path, and introduced as a minor-version addition. Consumers reading only `0` / non-zero continue to work across any such addition.

Problems: none directly — aligns the SDK with the GNU / POSIX / clap / cobra / argparse convention. Starting narrow preserves additive room; every exit code is a contract commitment (Hyrum's Law) so the v1 set is deliberately minimal.

#### S2-19: Platform Matrix

The SDK v1 must ship pre-built binaries for exactly four targets:

- `x86_64-unknown-linux-gnu`
- `aarch64-unknown-linux-gnu`
- `aarch64-apple-darwin`
- `x86_64-pc-windows-gnu`

The matrix mirrors `influxdb_pro`'s release matrix exactly so that the SDK's release pipeline can reuse the org's existing cross-builder image (`us-east1-docker.pkg.dev/influxdata-team-edge/ci-support/ci-cross-influxdb3`) without forking or extending its toolchain set. Linux binaries dynamically link glibc; macOS and Windows binaries have no non-OS runtime dependencies. Alpine Linux, Intel macOS, and Windows-on-ARM are out of scope for pre-built v1 binaries; users on those platforms install via `cargo install --git` (per [[#S2-20 v1 Distribution Channel|S2-20]]) or rebuild from source. Dropping any target from the matrix is a major-version change of `influxdb3-plugin-cli`; adding targets is minor-version additive.

Problems: none directly — the matrix covers the org's actual deployment footprint, and `cargo install --git` covers the long tail. Cross-builder image reuse is the explicit constraint that defines the matrix shape; widening it would either fork the image (significant maintenance burden) or split the release pipeline across multiple build environments.

#### S2-20: v1 Distribution Channel

GitHub Releases and `cargo install --git` from the SDK's source repository are the supported distribution channels for SDK v1. Each tagged release must publish, to GitHub Releases:

- One `influxdb3-plugin-<target-triple>.tar.gz` per target from [[#S2-19 Platform Matrix|S2-19]] (tarball used for all targets including Windows, mirroring `influxdb_pro`'s release artifact convention).
- A `SHA256SUMS` file covering every asset.
- Release notes.

`cargo install` consumes the same tagged commits via `cargo install --git https://github.com/influxdata/influxdb3-plugin-sdk --tag vX.Y.Z influxdb3-plugin-cli`, building from source. The four-layer SHA precedence in [[#S2-21 --version Output Shape|S2-21]] ensures `cargo install` builds carry an authoritative revision: for `--git` installs, Cargo preserves `.git` metadata in its checkout, so the `git rev-parse HEAD` layer (layer 3) fires. For the future crates.io install path (post-Group-H, gated on security + legal review), the `.cargo_vcs_info.json` layer (layer 2) fires instead — that file is generated at `cargo publish` time and ships inside the published `.crate` tarball.

Publishing the SDK crates to crates.io is deferred until security and legal review of the crate's publication completes. Once approved, `cargo install influxdb3-plugin-cli` (without `--git`) becomes the third v1 distribution channel; the change is purely a `publish = false → true` flip per crate plus a one-time crates.io publication. The deferred publication does not change S2-20's contract — `cargo install --git` already satisfies the consumer use case for users with internal-repo access (the dominant audience while the SDK is internal-only).

Consumers verify integrity via SHA-256 in v1 for GitHub Releases binaries; `cargo install --git` integrity is anchored by Git's commit SHA at the tagged ref. Cryptographic signing and additional distribution channels (Homebrew tap, winget, container images, OS-native packages) are planned post-v1 and additive — they layer on top of the v1 channels, not in place of them.

Problems: none directly — this sets the v1 canonical channels so downstream tooling has a stable target. The two v1 channels (GitHub Releases binaries, `cargo install --git`) cover both binary-consumption (CI/CD agents, dev laptops with no Rust toolchain) and source-build (Rust developers, platforms outside the [[#S2-19 Platform Matrix|S2-19]]) use cases.

#### S2-21: `--version` Output Shape

The top-level `--version` flag must print exactly one line in the shape:

```
influxdb3-plugin <version>, revision <sha>
```

- `<version>` is the SDK's full SemVer string from `CARGO_PKG_VERSION`, including any pre-release identifiers (e.g., `3.9.0-1.rc.0`) and build metadata when present.
- `<sha>` is the 40-character lowercase-hex git commit hash from which the binary was built, or the literal string `unknown` when no source produces a value.

The format aligns with the `influxdb3` binary's `build_version_string` (`{product}, {version}, revision {sha}`, defined in `oss/influxdb3_process/src/lib.rs` in the `influxdata/influxdb_pro` repo) so that when the SDK is embedded as `influxdb3 plugin --version` per [[#S2-5 Self-Declared Version|S2-5]], the embedded subcommand's output is visually consistent with the host's top-level `influxdb3 --version`.

The `<sha>` value is sourced by the following precedence, evaluated at compile time:

1. The `GIT_HASH` environment variable, when set non-empty. Lets release pipelines bake an authoritative SHA from CI's checkout.
2. The `git.sha1` field of `.cargo_vcs_info.json` at `$CARGO_MANIFEST_DIR`. This file is written by `cargo publish` and ships inside the published `.crate` tarball; reading it identifies the published artifact's commit even when the consumer has no `.git` directory (the future crates.io `cargo install` case, post-Group-H). For the v1 `cargo install --git` path, this layer does not fire — Cargo preserves `.git` metadata in `--git` checkouts, so layer 3 (`git rev-parse HEAD`) is what actually returns the SHA.
3. `git rev-parse HEAD` from the build environment. The local-development case.
4. The literal `unknown`. Final fallback for genuinely uncontrolled rebuilds — Homebrew, Nix, hand-unpacked tarballs rebuilt outside `cargo install`. Preserves the one-line overall shape so downstream regex parsers do not need to branch.

The format is stable across minor-version bumps so CI scripts can regex it; changes to the format or to the precedence chain are a major-version change of `influxdb3-plugin-cli`.

`--version` is exempt from [[#S2-13 Output Mode Flag and Defaults|S2-13]] — it emits plain text regardless of `--output` mode, matching universal CLI convention.

A build date is intentionally not included. Per-build dates would defeat binary reproducibility (same source + different wall-clock time = different binary bytes); the SHA alone is sufficient for traceability.

Problems: none directly — supports bug-report attribution and CI-side reproducibility verification (e.g., asserting a specific commit is running in a protected release pipeline).
